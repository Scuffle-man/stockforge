**本案核心表设计（带注释）**：
```

- created_at

inventory (库存表，记录每一次入库或形态变化后的批次)

- id, original_transaction_id, sn, return_reason, cost_impact
- created_at
```
**为什么存成本在批次表？** 因为价格会变，每次入库时的成本需要固化，计算时用移动平均或先进先出。

## 一、项目代码的文件结构（Go 标准布局）

```
warehouse/

├── pkg/                    # 可被外部引用的公共库 (可选)

├── config/                 # 配置文件模板

├── Makefile                # 常用命令快捷方式
```

**为什么这样分？**
- `cmd/` 与 `internal/` 分离：防止你将业务代码写在入口文件里，保证可测试、可替换。
- `handler -> service -> repository` 分层：**单向依赖**，上层只依赖下层，不得反向调用。这样改数据库只需动 repository，改业务只动 service。
- `migrations/` 独立：数据库结构必须**版本化**，和生产环境同步，不能手工在数据库里直接改表。

## 二、每层代码的标准化写法（带注释讲为什么）

### 1. 模型层 (`internal/model/product.go`)
```go
type Product struct {
    ID        int64     `db:"id" json:"id"`
    SN        string    `db:"sn" json:"sn"`               // 唯一索引
    Model     string    `db:"model" json:"model"`
    Category  string    `db:"category" json:"category"`   // 整机/零件
    CreatedAt time.Time `db:"created_at" json:"created_at"`
}
// 字段标签 db 用于 sqlx 扫描，json 用于输出，**不要**混入 ORM 复杂关联
```
**为什么直接用 db 标签而不是 GORM 的 Belongs To？**  
保持模型纯粹，复杂关联应显式在 service 层用多个 repo 调用组合，性能可控且易于理解。

### 2. 仓库层 (`internal/repository/product_repo.go`)
```go
type ProductRepo struct {
    db *sqlx.DB
}

func NewProductRepo(db *sqlx.DB) *ProductRepo {
    return &ProductRepo{db: db}
}

// GetBySN 查单条，返回 model 和 error
func (r *ProductRepo) GetBySN(ctx context.Context, sn string) (*model.Product, error) {
    var p model.Product
    err := r.db.GetContext(ctx, &p, "SELECT * FROM product WHERE sn=?", sn)
    if err != nil {
        if errors.Is(err, sql.ErrNoRows) {
            return nil, ErrNotFound  // 自定义错误，上层能区分“没找到”和“数据库挂了”
        }
        return nil, fmt.Errorf("query product by sn: %w", err)
    }
    return &p, nil
}
// 所有 repo 方法必须接受 context，用于超时控制和链路追踪。
```
**为什么用错误包装 (`%w`)？** 上层可以用 `errors.Is` 判断原始错误，日志也能打印完整链路。

### 3. 服务层 (`internal/service/product_svc.go`)
```go
type ProductSvc struct {
    productRepo *repository.ProductRepo
    invRepo     *repository.InventoryRepo
}

// Inbound 入库业务，需要事务
func (s *ProductSvc) Inbound(ctx context.Context, req *model.InboundReq) error {
    // 业务校验：SN 不能重复
    _, err := s.productRepo.GetBySN(ctx, req.SN)
    if err == nil {
        return errcode.ErrSNDuplicate
    }
    if !errors.Is(err, ErrNotFound) {
        return err
    }

    // 开启事务（简化，实际应使用 db.BeginTx）
    tx, _ := s.productRepo.BeginTx(ctx)
    defer tx.Rollback()

    // 创建产品
    product := &model.Product{SN: req.SN, Model: req.Model}
    if err := s.productRepo.CreateWithTx(ctx, tx, product); err != nil {
        return err
    }
    // 创建初始库存
    inventory := &model.Inventory{
        ProductID: product.ID,
        SN:        req.SN,
        UnitCost:  req.Cost,
        Type:      "入库",
    }
    if err := s.invRepo.CreateWithTx(ctx, tx, inventory); err != nil {
        return err
    }
    return tx.Commit()
}
```
**为什么要有事务？** 入库既要写产品表又要写库存表，必须同时成功或失败，否则数据会脏。  
**为什么用自定义错误码？** 让 handler 层方便返回对应 HTTP 状态码。

### 4. 处理器层 (`internal/handler/product.go`)
```go

// 依赖注入通过构造函数传入，不用全局变量

    return &ProductHandler{svc: svc}
}

func (h *ProductHandler) Inbound(c *gin.Context) {
    var req model.InboundReq
    if err := c.ShouldBindJSON(&req); err != nil {
        response.Error(c, errcode.ErrInvalidParam, err.Error())
        return
    }
    if err := h.svc.Inbound(c.Request.Context(), &req); err != nil {
        // 根据错误类型返回不同状态码
        response.Error(c, errcode.ErrCode(err), err.Error())
        return
    }
    response.Success(c, nil)
}
```
**为什么 handler 这么薄？** 它只做参数绑定和响应，绝不包含业务，因为 HTTP 框架可能替换（Gin 换 Echo），业务逻辑不变。

### 5. 中间件：统一日志和恢复 (`internal/middleware/logger.go`)
```go
func Logger() gin.HandlerFunc {
    return func(c *gin.Context) {
        start := time.Now()
        c.Next()
        latency := time.Since(start)
        log.Printf("[%d] %s %s %v", c.Writer.Status(), c.Request.Method, c.Request.URL.Path, latency)
    }
}
```
**为什么要有恢复中间件？** 防止某个 panic 导致整个程序崩溃，能记录堆栈并返回 500。

## 三、更深层的工程化实践

### 1. 依赖注入与组装（`cmd/server/main.go`）
```go
func main() {
    // 1. 加载配置
    cfg := config.Load("config/config.yaml")

    // 2. 初始化数据库连接
    db, err := sqlx.Connect("mysql", cfg.DSN)
    if err != nil {
        log.Fatal(err)
    }

    // 3. 初始化各层（手动依赖注入）
    productRepo := repository.NewProductRepo(db)
    productSvc := service.NewProductSvc(productRepo, inventoryRepo)
    productHandler := handler.NewProductHandler(productSvc)

    // 4. 设置路由
    r := gin.Default()
    r.Use(middleware.Logger(), middleware.Recovery())
    api := r.Group("/api/v1")
    {
        api.POST("/products/inbound", productHandler.Inbound)
    }

    // 5. 启动服务，含优雅关闭
    srv := &http.Server{Addr: ":8080", Handler: r}
    go func() {
        if err := srv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            log.Fatal(err)
        }
    }()
    // 等待退出信号...
}
```
**为什么需要优雅关闭？** 收到 SIGTERM 时，先停止接收新请求，等待现有请求处理完，再关闭数据库连接，避免数据丢失。

### 2. 配置管理
不要将密码写死在代码里，使用环境变量或 yaml 文件。
```yaml
# config/config.yaml
server:
  port: 8080
database:
  host: ${DB_HOST:localhost}
  port: ${DB_PORT:3306}
  user: ${DB_USER:root}
  password: ${DB_PASSWORD:secret}
```
**为什么用环境变量覆盖？** 容器化部署时只需注入环境变量，不需要改配置文件。

### 3. 数据库迁移
使用 `golang-migrate`，通过 CLI 执行：
```
migrate -path migrations -database "mysql://user:pass@tcp(localhost:3306)/warehouse" up
```
**为什么严禁手工改表？** 测试库、生产库的结构差异会引发巨大故障，所有 DDL 必须代码化、可回滚。

### 4. 错误码标准化
```go
package errcode

var (
    ErrSNDuplicate  = New(10001, "SN已存在")
    ErrNotFound     = New(10002, "数据不存在")
    ErrInternal     = New(50001, "服务器内部错误")
)
type ErrCode struct { Code int; Msg string }
```
**为什么要有错误码？** 前端可以根据 code 做不同处理，且国际化方便。

### 5. 单元测试与 Mock
service 层测试时，需要 mock repository，使用接口+手工 mock 或 `gomock`。
```go
type ProductRepoInterface interface {
    GetBySN(ctx context.Context, sn string) (*model.Product, error)
    CreateWithTx(...)
}
```
**为什么 repository 要定义接口？** 使得 service 依赖接口而非具体实现，测试时可注入 mock 对象，避免连真实数据库。

### 6. Docker 多阶段构建
```dockerfile
FROM golang:1.20-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o server ./cmd/server

FROM alpine:3.18
RUN apk add --no-cache ca-certificates
COPY --from=builder /app/server /server
COPY --from=builder /app/config /config
EXPOSE 8080
CMD ["/server"]
```
**为什么多阶段？** 构建阶段需要 Go 环境，运行阶段只需一个空镜像加可执行文件，最终镜像极小（约15MB）。

### 7. Makefile 管理常用命令
```makefile
.PHONY: run build test migrate-up
run:
    go run ./cmd/server
build:
    docker build -t warehouse:latest .
test:
    go test ./internal/...
migrate-up:
    migrate -path migrations -database "mysql://root:secret@tcp(localhost:3306)/warehouse?multiStatements=true" up
```
**为什么需要 Makefile？** 统一团队操作入口，新人不用记住长长的命令。

## 四、你从设计到实现的标准动作流程（按顺序）

1. 用 `go mod init warehouse` 初始化项目。
2. 按上述结构创建目录和空文件。
3. 先定义 `model` 结构体和 `repository` 接口。
4. 编写 `migrations` 文件，创建初始表结构并执行。
5. 实现 `repository` 的具体 SQL 方法，写单元测试验证数据库连接。
6. 实现 `service` 层，用 mock repository 写测试。
7. 实现 `handler`，注册路由，用 `httptest` 写接口测试。
8. 编写 `middleware`（日志、恢复）。
9. 完善 `main.go` 组装，启动服务。
10. 本地用 curl 或 Postman 验证接口。
11. 编写 Dockerfile 和 docker-compose.yml，用 docker-compose up 一键启动全套。
12. 继续迭代新功能，严格遵循“迁移 -> repo -> service -> handler”的顺序。

## 五、再补充几个“更深层”的要点

- **长事务处理**：如果拆改明细更新多张表，一定要控制事务时长，避免锁表。
- **SN 唯一性**：不要仅依赖数据库唯一索引，service 层也要显式校验，减少无谓的数据库错误。
- **成本计算**：如果是移动平均成本，需要写一个专门的方法，接收出入库事件，依次计算并更新库存均价。
- **并发安全**：针对同一 SN 的并发操作，可以利用数据库行锁（SELECT ... FOR UPDATE）或分布式锁来防止超卖/超拆。
- **日志规范**：使用结构化日志（如 `zap`），每条日志携带 trace_id，便于问题排查。
- **健康检查**：提供一个 `/health` 端点，返回 DB 连接状态，用于 K8s 或 Docker 探活。

**最后，理解你“做过实施但没亲手从头写”的痛点**：实施通常拿到的是一套成型系统，你少了从空白文件夹到跑通第一个接口的过程。现在用这个“骨架”去新建一个仓库，按照我列的动作1到12走一遍，哪怕接口只返回 `{"message":"ok"}`，你也算真正跨过了从0到1的门槛。边做边查，你缺的不是知识，是“执行一次”的经验。

## 如何实现“分岗位，拿到即用”？

关键在于 **README.md** 和 **Makefile** 构成的总入口，让任何人只需看这两处就能找到自己需要的。

### 1. README.md 要分角色写

```markdown
# 仓库管理系统

## 项目简介
电脑拆改、成本、退货、SN管理后端服务。

## 角色指南
### 开发人员
- 快速启动：`make dev`
- 运行测试：`make test`
- 查看接口文档：启动后访问 http://localhost:8080/swagger/index.html

### 测试人员
- 环境搭建：参照 `docs/setup.md` 用 Docker 起环境
- 测试用例：查看 `docs/api.md` 和 Postman 集合链接（或放在 docs 下）
- 执行自动化测试：`make test-integration`

### 实施/部署人员
- 生产部署：`make deploy-prod` 或参考 `docs/deploy.md`
- 数据库更新：`make migrate-prod`
- 回滚：`make rollback`

### 运维人员
- 健康检查端点：`GET /health`
- 日志查看：`docker logs -f warehouse`
- 监控指标：`GET /metrics` (如集成 Prometheus)
- 故障处理：见 `docs/troubleshooting.md`
```

### 2. Makefile 作为统一操作面板

把所有手动操作都写成 make 目标，消除复制粘贴。

```makefile
.PHONY: help dev test build migrate-up migrate-down deploy-prod clean

## help: 显示所有命令说明
help: Makefile
	@sed -n 's/^##//p' $<

## dev: 本地开发启动（用 Docker 起 MySQL，Go 热加载）
dev:
	docker-compose -f deployments/docker-compose.yml up -d mysql
	air -c .air.toml

## test: 运行单元测试
test:
	go test ./internal/... -count=1

## test-integration: 运行集成测试（需数据库）
test-integration:
	go test ./test/integration/... -tags=integration

## build: 编译生产二进制
build:
	CGO_ENABLED=0 go build -o bin/server ./cmd/server

## migrate-up: 执行数据库升级
migrate-up:
	migrate -path migrations -database "$(DATABASE_URL)" up

## migrate-down: 回滚最近一次迁移
migrate-down:
	migrate -path migrations -database "$(DATABASE_URL)" down 1

## deploy-prod: 生产环境部署（Docker Compose）
deploy-prod:
	docker-compose -f deployments/docker-compose.prod.yml up -d --build

## clean: 清理构建产物和本地容器
clean:
	docker-compose -f deployments/docker-compose.yml down -v
	rm -rf bin/
```

**为什么这么做？** 任何人下载代码后，只需 `make help` 就知道所有能做的事，不必问人。

## 不同角色工作流详解

### 开发人员拿到代码后

1. 克隆仓库，执行 `make dev`。
2. `make dev` 会自动下载依赖、用 Docker 启动 MySQL，并通过 `air` 热加载 Go 代码。
3. 改了 SQL？写一个迁移文件到 `migrations/`，然后 `make migrate-up`。
4. 要加接口？在 `internal/handler` 里加函数，用 swaggo 注释生成文档。
5. 提交代码前运行 `make test`，通过后提 PR。

### 实施/部署人员

他们只看 `deployments/` 和 `docs/deploy.md`：
- 生产环境用 `docker-compose.prod.yml`（映射宿主机端口、数据卷、环境变量），执行 `make deploy-prod`。
- 数据库变更，由 DBA 审查 migrations 文件，上线时执行 `make migrate-up`。

### 测试人员

他们完全不看代码，而是用：
- 接口文档（启动后 swagger UI）或 `docs/api.md`。
- `scripts/seed.sh` 灌入测试数据。
- Postman 或自动化脚本覆盖核心流程。

### 运维人员

- 容器挂了自动重启？已在 `docker-compose.prod.yml` 设置 `restart: always`。
- 查看日志：`docker logs -f`，日志输出为 JSON 结构化（配置在代码中）。
- 更新应用：拉取新镜像后 `docker-compose up -d`。
- 数据库备份：另写脚本在 `scripts/backup.sh` 并加入定时任务。

## 针对“不能什么都手动复制粘贴”的自动化手段

### 数据库建表与更新
使用 **golang-migrate**（或其他迁移工具），表结构永不手动操作。
- 新建变更：`migrate create -ext sql -dir migrations -seq add_return_reason`
- 生成两个文件：`000002_add_return_reason.up.sql` 和 `.down.sql`
- 所有开发环境执行 `make migrate-up`，CI/CD 自动跑迁移测试。

### 配置管理
不同环境（开发、测试、生产）配置值不同，但不改变结构。
- 代码里只读一个结构体，值从环境变量注入：
  ```yaml
  database:
    host: ${DB_HOST:localhost}
  ```
- 实施人员修改 `.env` 文件或 K8s ConfigMap，不碰代码。

### 接口文档自动生成
在 handler 里写 swaggo 注释，构建时运行 `swag init` 生成 docs，并挂载到服务。开发修改接口后只需重新生成，文档永远一致。

### 自动化测试环境
CI 文件 (`ci.yml`) 在每次推送时：
1. 启动 MySQL 容器
2. 执行 `make migrate-up`
3. 执行 `make test`
4. 执行 `make test-integration`
5. 构建镜像并推送到仓库

这样任何人提 PR 都能看到测试结果，保障不引入破坏性变更。

## 为什么这就是大厂标准化？

- **代码与基础设施分离**：业务代码完全不感知部署方式（Docker/K8s/物理机），只通过环境变量获取配置。
- **所有操作代码化**：一切手动步骤都写成脚本或 Makefile，纳入版本控制，可审计、可复现。
- **按角色拆分入口**：不用让运维看 Go 代码，也不用让开发管 Nginx 配置，但仓库里都有明确位置。
- **自动化门禁**：测试、格式检查、安全扫描在 CI 流程里强制执行。
