## 一、设计思路与优化合并

1. **SN 作为唯一资产标识**  
    所有机器、配件、显示器均以 `sn` 作为全局唯一标识。`items` 表是“物品主表”，一条记录对应一个 SN，记录该物品的固有属性、当前状态、成本等。库存是否在库、去向如何，都通过该表的 `status` 和关联字段体现。
    
2. **库存查询由视图支撑**  
    前端需要的“按品牌/型号/CPU/内存/固态/WIFI 搜索，显示数量和总价”是**汇总视图**，不是直接查 SN 明细。因此用 `products` 表定义标准配置（SKU），再通过 `items.status = 'in_stock'` 聚合得到每种配置的当前库存数量和总成本。库存预警也放在 `products` 表中。
    
3. **采购入库简化**  
    设计 `purchase_orders`（采购单头）和 `purchase_inbound_items`（扫码 SN 明细）。总数量校验、SN 正确性校验在后端通过比对 `expected_qty` 与实际扫码数量完成，表里只存结果。
    
4. **入仓与厂直统一为 `fulfillments` 表**  
    两种业务共用同一张主表，用 `type` 字段区分（`inbound` / `direct`）。特有字段允许为空。拆改明细统一用子表 `disassembly_details`，精确到 SN、价格、动作（加/拆）、操作人。工资根据 `disassembly_details` 关联的 `salary_rules` 或直接记录计算。
    
5. **备件库退货与京东退货统一为 `returns` 表**  
    通过 `source_type` 区分来源（备件库退/直寄退/京东退货），SN 必须关联。检验结果、处理方式、前后 SN 变更都在此表处理。状态联动通过后端更新 `items` 表实现。
    
6. **返修配件独立 `repairs` 表**  
    记录送修 SN、返回 SN、维修状态，并可通过 `alert_on_stocktake` 在盘点时高亮提示。修理后出库二手或报废，会同步更新 `items` 状态。
    
7. **所有变动可追溯**  
    设计 `inventory_transactions` 流水表，记录每次库存变化（入库、出库、拆改消耗、拆改产出、退货、返修等），方便对账。

## 三、关于前端搜索与库存预警

- **库存视图**：为方便前端直接使用，可创建数据库视图，例如：
    

sql

CREATE VIEW v_inventory_summary AS
SELECT 
    p.id AS product_id,
    p.product_name,
    p.brand,
    p.model,
    p.category,
    p.cpu,
    p.memory,
    p.storage,
    p.gpu,
    p.wifi,
    COUNT(i.sn) AS stock_qty,
    SUM(i.unit_cost) AS total_cost,
    p.min_stock
FROM products p
LEFT JOIN items i ON i.product_id = p.id AND i.current_status = 'in_stock'
GROUP BY p.id;

前端可对此视图进行任意组合搜索，并比对 `stock_qty` 与 `min_stock` 实现预警。

## 优化后的前端接口规划（RESTful API）

### 1. 库存相关（前端最核心的搜索/统计）

- **产品库存汇总（前台搜索用）**  
    `GET /api/inventory/products/summary`  
    参数：`?brand= &model= &category= &cpu= &memory= &storage= &gpu= &wifi= &keyword= &min_stock_alert=true`  
    返回：按产品配置聚合的库存数量、总成本、是否低于预警阈值。  
    对应数据库视图 `v_inventory_summary`。
    
- **SN 明细列表（点击某个产品下钻）**  
    `GET /api/inventory/items`  
    参数：`?product_id= &status=in_stock &brand= &cpu= &sn= &page= &limit=`  
    返回：分页的单个 SN 记录，用于盘库或追踪。
    
- **库存全局统计**  
    `GET /api/inventory/statistics`  
    返回：在库总价值、各类别（整机/配件/显示器）数量、库存预警品类数等。
    
- **库存预警列表**  
    `GET /api/inventory/alerts`  
    返回：所有 `current_stock < min_stock` 的产品配置及缺货数量。
    

### 2. 采购入库

- **采购单管理**  
    `GET /api/purchase-orders`（列表，可按状态/日期筛选）  
    `POST /api/purchase-orders`（创建采购单，含预期数量和供应商）  
    `GET /api/purchase-orders/{id}`（采购单详情，含预期总数量）
    
- **扫码入库（SN明细）**  
    `POST /api/purchase-orders/{id}/items`  
    Body：`{ "sn": "...", "product_name": "...", "category": "...", "unit_price": ... }`  
    后端写入 `purchase_inbound_items` 并自动创建对应的 `items` 记录（若 SN 不存在）。
    
- **入库校验**  
    `POST /api/purchase-orders/{id}/verify`  
    后端计算已扫有效 SN 数量，与采购单预期数量对比，更新 `match_result` 字段，并返回校验结果（一致/短缺/溢出）。
    

### 3. 入仓/厂直出库与拆改（对应 fulfillments）

- **出库单管理**  
    `GET /api/fulfillments?type=inbound&status= &date=`（入仓）  
    `GET /api/fulfillments?type=direct&status= &date=`（厂直）  
    `POST /api/fulfillments`  
    Body 中包含 type 及其他对应字段（入仓填采购单号等，厂直填订单号、客户信息等）。
    
- **提交拆改明细**  
    `POST /api/fulfillments/{id}/disassembly`  
    Body：`[{ "machine_sn":"...", "action":"add", "part_sn":"...", "part_price":..., "staff":"..." }, ...]`  
    后端写入 `disassembly_details`，同时更新配件 `items.status` = `used_in_modification`，并将配件的 `assembled_into_sn` 指向机器 SN。
    
- **标记拆改完成**  
    `PUT /api/fulfillments/{id}/complete`  
    更新 `is_disassembly_completed = true`，并校验 SN 数量是否匹配实际送货数量。
    

### 4. 工资结算

- **工资规则管理**  
    `GET /api/salary-rules`（查看入仓/厂直单台拆改价格）
    
- **工资记录**  
    `GET /api/salary/records?staff= &date= &fulfillment_type= &is_paid=`  
    `POST /api/salary/generate`（可选：根据已完成拆改的出库单自动生成工资记录，也可手动逐条创建）
    

### 5. 退货/备件库处理（统一 returns）

- **退货单管理**  
    `GET /api/returns?source_type=spare_parts&status=`  
    `POST /api/returns`（创建退货单，包含原 SN 等信息）
    
- **处理退货**  
    `PUT /api/returns/{id}/handle`  
    Body：`{ "inspection_result": "...", "handling_method": "...", "new_sn": "...", "is_completed": true }`  
    后端同步更新 `items` 中对应 SN 的状态（原 SN 变为 `returned`，新 SN 变为 `in_stock` 等）。
    

### 6. 返修配件

- **返修单**  
    `GET /api/repairs?status=sent&alert=true`（待修回列表，前端可高亮）  
    `POST /api/repairs`（送修登记）  
    `PUT /api/repairs/{id}`（修回更新，填写 `returned_sn`、`disposal_method`，同时更新 items 状态为 `in_stock` 或 `scrapped`）
    

### 7. 财务成本核算

- **成本汇总**  
    `GET /api/finance/cost-summary?start_date= &end_date= &status=sold,scrapped,used_in_modification`  
    返回：各状态下物品的总成本，可按月份或类别分组。

func GetProductStockSummary(c *gin.Context) {
    type StockSummary struct {
        ProductID   uint    `json:"product_id"`
        ProductName string  `json:"product_name"`
        Brand       string  `json:"brand"`
        Model       string  `json:"model"`
        Category    string  `json:"category"`
        CPU         string  `json:"cpu"`
        Memory      string  `json:"memory"`
        Storage     string  `json:"storage"`
        GPU         string  `json:"gpu"`
        WIFI        string  `json:"wifi"`
        StockQty    int     `json:"stock_qty"`
        TotalCost   float64 `json:"total_cost"`
        MinStock    int     `json:"min_stock"`
    }
    var summaries []StockSummary
    query := database.DB.Table("products p").
        Select(`p.id as product_id, p.product_name, p.brand, p.model, p.category,
                p.cpu, p.memory, p.storage, p.gpu, p.wifi,
                COUNT(i.sn) as stock_qty,
                COALESCE(SUM(i.unit_cost),0) as total_cost,
                p.min_stock`).
        Joins("LEFT JOIN items i ON i.product_id = p.id AND i.current_status = ?", "in_stock").
        Group("p.id")
    // 搜索条件（示例）
    if brand := c.Query("brand"); brand != "" {
        query = query.Where("p.brand = ?", brand)
    }
    if cpu := c.Query("cpu"); cpu != "" {
        query = query.Where("p.cpu = ?", cpu)
    }
    if keyword := c.Query("keyword"); keyword != "" {
        query = query.Where("p.product_name LIKE ?", "%"+keyword+"%")
    }
    // 是否只显示缺货
    if c.Query("min_stock_alert") == "true" {
        query = query.Having("stock_qty < p.min_stock")
    }
    err := query.Scan(&summaries).Error
    if err != nil {
        c.JSON(http.StatusInternalServerError, gin.H{"error": err.Error()})
        return
    }
    c.JSON(http.StatusOK, summaries)
}

- 拆改明细接口 `AddDisassembly`：接收 `[{machine_sn, action, part_sn, part_price, staff}]`，写入 `disassembly_details`，同时更新 `items` 表中配件的状态（`used_in_modification`）和 `assembled_into_sn` 指向机器 SN。这些在事务中完成确保一致性。

// AddDisassembly 示例伪代码
func AddDisassembly(c *gin.Context) {
    fid := c.Param("id")
    var details []struct {
        MachineSN  string  `json:"machine_sn" binding:"required"`
        Action     string  `json:"action" binding:"required"` // add/remove
        PartSN     string  `json:"part_sn" binding:"required"`
        PartPrice  float64 `json:"part_price"`
        Staff      string  `json:"staff"`
    }
    if err := c.ShouldBindJSON(&details); err != nil {
        c.JSON(400, gin.H{"error": err.Error()})
        return
    }
    tx := database.DB.Begin()
    for _, d := range details {
        dis := models.DisassemblyDetail{
            FulfillmentID: fidConvert,
            MachineSN:     d.MachineSN,
            Action:        d.Action,
            PartSN:        d.PartSN,
            PartPrice:     d.PartPrice,
            StaffName:     d.Staff,
        }
        if err := tx.Create(&dis).Error; err != nil {
            tx.Rollback()
            c.JSON(500, gin.H{"error": err.Error()})
            return
        }
        // 更新物品状态
        if d.Action == "add" {
            tx.Model(&models.Item{}).Where("sn = ?", d.PartSN).
                Updates(map[string]interface{}{
                    "current_status":    "used_in_modification",
                    "assembled_into_sn": d.MachineSN,
                })
        } else {
            // remove 操作：配件回到库存，清除组装关系
            tx.Model(&models.Item{}).Where("sn = ?", d.PartSN).
                Updates(map[string]interface{}{
                    "current_status":     "in_stock",
                    "assembled_into_sn":  "",
                })
        }
    }
    tx.Commit()
    c.JSON(200, gin.H{"message": "success"})
}

## 一、核心设计原则：让数据库说真话，让代码只做翻译

整个系统的本质是：**真实世界发生了一件事（到货、拆改、退货），你要在数据库里留下一笔不会说谎的记录，然后任何时候都能查出当前库存和成本。**

所以设计上抓住一条主线：**以 `items` 表为唯一真相来源（Single Source of Truth）**。  
所有业务动作（采购、拆改、退货、返修）最终都是修改 `items` 表中某个 SN 的 `current_status` 和去向字段。

这意味着：

- 前端显示库存时，不必去猜“这个配件现在在哪”，只要查 `items` 表 `current_status = 'in_stock'` 就一定是库里的。
    
- 对账、盘点、财务成本核算，都只信 `items` 表的状态和成本字段。
    
- 业务表（采购单、出库单、退货单）只是“过程证据”，负责记录谁在什么时候做了什么，但它们不能绕过 `items` 表直接改变库存认知。
    

---

## 二、表结构设计的推导过程（为什么这样建表）

你之前的原始需求很长，我把它翻译成三个层次：

### 1. 物品本身：`products` + `items`

- **为什么拆成两张表？**  
    `products` 描述“标准配置”，比如“ThinkPad X1 Carbon i7/16G/512G”。它是一个模板，用来快速创建物品和做库存汇总（按型号查库存量）。  
    `items` 是实物，每个 SN 一条记录。因为每个 SN 的命运不同：有的在库，有的被拆了，有的送修了。如果只用一张表存 SN，当你要问“i7 的机器还有多少库存”时，得在几万条 SN 里去做 `WHERE cpu='i7' AND status='in_stock'`，既慢又不准（CPU 字段可能写错）。有了 `products` 抽象，汇总查询就很快，而且能让同一型号的不同 SN 都关联到同一个产品 ID。
    

### 2. 业务过程：`purchase_*`, `fulfillments`, `returns`, `repairs`

- **采购入库为什么分 `purchase_orders` + `purchase_inbound_items`？**  
    因为采购单是一个“计划”，实际到货可能分批、可能 SN 扫错。分开后，你可以看“订单应该到了 10 个，今天扫了 8 个”，而不污染采购单本身。扫码时直接写入明细表，校验接口只做比较，不修改采购单头。  
    而且 `items` 表里会同步创建 SN 记录（如果新 SN 出现的话），保证 SN 的唯一性。
    
- **入仓和厂直为什么合成一张 `fulfillments` 表？**  
    你需求里说“如果可能，合成一个表，前端无差异”。虽然它们业务背景不同（入仓是送京东仓，厂直是发给客户），但它们都是“要发出一批机器，可能涉及拆改，最后要算工资”。合并成一张表后，只需要用 `type` 字段区分，共用拆改明细子表（`disassembly_details`），共用工资生成逻辑。前端展示时，只要传 `type` 参数就能切换视图。
    
- **拆改明细为什么要单独一张 `disassembly_details` 表？**  
    因为拆改不是简单的一个备注字段，它要精确到“从哪台机器（SN）上拆下了哪块固态（SN），加到了哪台机器上”，而且每个动作的成本要能算进工资。如果用一个大 JSON 字段塞在 `fulfillments` 里，将来你想查“张三这个月加了多少根内存”几乎不可能。独立子表后，查工资、算配件成本、追溯拆改历史都很方便。
    
- **退货为什么统一成 `returns` 表？**  
    三种退货（备件库、京东仓、直寄退）的共同点是：都要关联原 SN，都要记录检验结果和处理方式，而且处理后必须同步修改 `items` 表的状态。合并后用 `source_type` 区分，处理逻辑高度复用，减少重复代码。
    
- **返修为什么独立一张表？**  
    因为返修是一个耗时流程（送修→修回），而且需要在盘点时特别提示。`repairs` 表里 `status='sent'` 的记录就代表“东西虽然不在库，但理论上还是我的资产”，盘库时查一下这张表高亮显示，避免以为丢了。
    

### 3. 财务和工资：`salary_*` + 直接查 `items` 成本

- 财务要“机器配件成本”，不需要额外表。所有 SN 的成本都在 `items.unit_cost` 里，查询时按状态过滤即可。比如 `current_status IN ('sold','scrapped','used_in_modification')` 就是已出库的成本总和。
    
- 工资表 `salary_records` 依赖于 `fulfillments` 和 `disassembly_details`。生成工资时，找已完成的拆改单，按规则（`salary_rules`）和人员记录生成，隔离了业务变更对工资的影响。

## 三、Go 代码的分层为什么这样组织（MVC 简化版）

代码结构不是随便分的，每一层解决一类问题：

text

请求 → routers（路由） → controllers（控制器） → models（数据定义） + database（数据库）

- **`models/`** ：就是数据库表的 Go 语言映射。你不用在代码里写 SQL 建表，GORM 自动完成。以后加字段改业务，先改这里的结构体。
    
- **`database/`** ：只做两件事：连接数据库、自动迁移。别的地方想操作数据库，直接 `database.DB.xxx`。
    
- **`controllers/`** ：处理 HTTP 请求，从请求里拿参数，调用 GORM 的查询/更新方法，返回 JSON。业务逻辑可以直接写在这里（小项目），或者抽到 `services/`（逻辑变复杂后）。
    
- **`routers/`** ：定义 URL 和控制器函数的对应关系。相当于一个“电话总机”，看到 `/api/inventory/items` 就转给 `GetItemList` 函数。
    
- **`middleware/`** ：处理跨域、日志这类所有请求都需要的通用功能。
    

**这样分层的好处**：当你想加一个“导出 Excel”功能时，只需要在 `controllers/` 里加一个函数，在 `routers` 里加一行路由，不会影响库存逻辑。改库存逻辑时只动 `controllers/` 里的几个函数，前端路由完全不用管。

## 四、核心业务流程是如何用 Go 代码实现的（设计意图）

### 1. 库存搜索为什么用联表查询而不是直接查 `items`？

因为你要的搜索维度（品牌、CPU、内存等）在 `products` 表里是标准化字段，而 `items` 表里虽然也有这些字段，但可能会有录入错误。我们用 `LEFT JOIN` 把 `products` 和状态为 `in_stock` 的 `items` 连接，按产品分组统计数量与总成本。这样：

- 即使某些产品还没有任何 SN，也会显示 0 库存（因为 LEFT JOIN）。
    
- 查询性能高，数据库只做一次聚合计算。
    
- 前端拿到的数据直接可渲染表格，不用自己加工。
    

### 2. 拆改业务的事务处理

拆改时如果只更新了 `disassembly_details` 而不更新 `items` 的状态，库存就乱了。所以必须用数据库事务：

- 开启事务 → 插入拆改明细 → 更新配件 SN 的 `current_status` 和 `assembled_into_sn` → 提交事务。
    
- 如果中间任何一步失败，整个回滚，不会出现“记录了加内存但配件状态还是未使用”的情况。  
    这是业务一致性的底线。
    

### 3. 工资计算的“触发”时机

工资不应该在创建拆改明细时立刻生成，因为可能拆改还没最终确认。所以我们给 `fulfillments` 一个 `is_disassembly_completed` 字段，只有标记完成后才生成工资。生成工资的接口可以手动调用（或定期任务），把该单下所有拆改动作按人员、规则汇总成 `salary_records`。这样财务才知道该给谁发多少钱。

### 4. 退货处理的“前后 SN 联动”

你的需求里说“需要改处理一个，另外的地方全变”。这本质是 **状态同步**。  
当退货处理接口 `PUT /api/returns/:id/handle` 被调用时，后端的逻辑是：

- 更新 `returns` 表的处理信息（检验结果、新 SN 等）
    
- 同步修改 `items` 表中原 SN 的状态为 `returned`
    
- 如果处理方式产生了新 SN（比如换良品），为新 SN 在 `items` 里创建一条 `in_stock` 记录  
    这样所有依赖 `items` 状态的查询自动生效，不存在“改了这里那里没变”。

# 库存管理系统整体设计文档

## 1. 项目概述

本系统用于管理计算机硬件（整机、显示器、内存、固态硬盘、显卡、WIFI模块等）的采购入库、SN序列号管理、入仓/厂直拆改、退货处理、返修追踪、库存盘点、成本核算及拆改工资计算。采用前后端分离架构，后端Go（Gin + GORM），前端React（Ant Design），数据库MySQL，局域网内可访问。

---

## 2. 核心设计原则

- **SN唯一性**：每个硬件（整机或配件）均以SN作为全局唯一标识，`items`表为库存状态的唯一真相来源。
    
- **状态同步**：任何业务动作（入库、拆改、退货、返修）必须同步更新`items`表的`current_status`，保证库存随时准确。
    
- **过程可追溯**：采购单、出库单、退货单、返修单等均为过程记录，不影响库存直接计算，但必须关联SN。
    
- **成本精确到SN**：每件物品单独记录成本（`unit_cost`），拆改时配件成本随动作转移，工资基于拆改明细计算。
    
- **前后端分离**：RESTful API，前端负责展示与交互，后端负责业务逻辑与数据校验。

## 4. 后端API设计（RESTful）

- **采购单CRUD**  
    `GET /purchase-orders`（列表）  
    `POST /purchase-orders`（创建）

- **扫码入库**  
    `POST /purchase-orders/{id}/items`

### 4.3 出库（入仓/厂直）模块

    `PUT /fulfillments/{id}/complete`

    Body: `{ "inspection_result","handling_method","new_sn","is_completed":true }`  
    处理时更新`items`：原SN状态改为`returned`；若提供了新SN，创建新物品记录（状态`in_stock`），并记录流水。
    
- **退货中的拆改**：退货检验时若需要拆配件，需调用拆改相关接口？ 可复用：在退货处理时如果涉及拆出配件，同样插入`disassembly_details`（可能需要一个关联退货ID的拆改表，或为退货也创建虚拟fulfillment？）  
    设计简化：退货处理的拆改也记录到`disassembly_details`，并增加`reference_type`和`reference_id`指向`returns.id`。修改`disassembly_details`结构，增加`reference_type`（可`fulfillment`/`return`）和`reference_id`，以适应退货拆改。同时更新`items`状态。
    

### 4.6 返修模块

- `GET /repairs?status=&alert=`
    
- `POST /repairs`
    
- `PUT /repairs/{id}`（修回更新，填写`returned_sn`、`disposal_method`，更新`items`状态）
    

### 4.7 财务模块

- **成本汇总**  
    `GET /finance/cost-summary?start_date=&end_date=&status=`（可多选状态）  
    统计`items`表中相应状态物品的成本总和。

## 6. 核心业务规则与流程

### 6.1 采购入库与校验

1. 创建采购单（预期数量、供应商等）。
    
2. 在采购单详情页扫码SN，每扫一个调用`POST /purchase-orders/:id/items`，后端：
    
    - 检查SN是否存在于`items`表，若不存在则创建（状态`pending`），关联`product_id`。
        
    - 写入`purchase_inbound_items`。
        
    - 前端展示已扫SN列表。
        
3. 点击校验：调用`POST /purchase-orders/:id/verify`，后端统计有效SN数，与`expected_total_qty`比对，更新`match_result`，并返回“匹配/短缺/溢出”。若匹配，可自动将`items`中对应SN状态改为`in_stock`（或手动确认后修改）。
    

### 6.2 出库（入仓/厂直）及拆改

1. 创建出库单（选择入仓或厂直，填写对应字段）。
    
2. 在出库单详情中，输入或扫描机器SN（多台），记录到出库单关联表（可增加`fulfillment_items`表存放出库SN，或先只在拆改时体现）。
    
    - 改进：增加`fulfillment_items`表，列出该单所有的机器SN（包括无需拆改的标机），并记录该机器拆改状态。
        
3. 对需要拆改的机器，在拆改组件中操作：
    
    - 加装配件：选择配件SN，系统自动带出配件成本（从`items.unit_cost`），执行后配件状态变为`used_in_modification`，`assembled_into_sn`指向机器SN。
        
    - 拆除配件：选择机器上已安装的配件（可通过查询`items` where `assembled_into_sn` = 机器SN），拆除后配件状态变回`in_stock`，清空组装关系。
        
    - 记录拆改明细（`disassembly_details`）和库存流水。
        
    - 机器本身状态在拆改过程中可保持`in_stock`，或标记为`pending_modification`。
        
4. 所有机器处理完毕，点击“完成拆改”，校验SN总数是否与`actual_delivery_qty`一致，更新出库单状态，并将所有机器SN状态改为`sold`（或根据业务实际出库状态）。
    

### 6.3 工资计算

- 根据`salary_rules`，入仓拆改和厂直拆改可能单价不同。
    
- 生成工资时，系统找出`is_disassembly_completed=true`且尚未生成工资记录（通过关联查询`salary_records`）的出库单，读取拆改明细中每个员工的操作记录，汇总生成工资记录。
    
- 工资记录可手动标记`is_paid`。
    

### 6.4 退货处理（含拆改）

1. 创建退货单，记录原SN。
    
2. 在退货处理详情中：
    
    - 填写检验情况。
        
    - 若需从退货机器上拆下配件，使用同样的拆改组件，但关联类型为`return`。拆下的配件恢复为`in_stock`（或根据检验结果定为`scrapped`）。
        
    - 记录拆改明细（`reference_type='return'`, `reference_id`）。
        
    - 处理方式选择后，更新原SN状态（如`returned`），若产生新SN（更换），创建新物品`in_stock`。
        
    - 处理完成标记`is_completed`。
        
3. 退货中的成本核算：拆下的配件成本回归库存，报废则成本计入损失（可在财务模块特殊处理）。
    

### 6.5 返修流程

1. 送修：创建返修单，SN状态改为`in_repair`，记录`alert_on_stocktake`=true。
    
2. 修回：填写`returned_sn`（可能换新SN），选择处置：二手出库（状态`sold`）或报废（`scrapped`）。若返回的SN与原SN不同，需创建新物品记录（状态相应），原SN根据处置改变状态。更新返修单`is_completed`。
    

### 6.6 成本核算

- 财务查询`items`表，按状态过滤：`sold`,`scrapped`,`used_in_modification`等，汇总`unit_cost`，可按月份（通过流水表或出入库时间）统计。

# 全栈库存管理系统 · 技术详解

## 第一部分：为什么选择 Go、Gin 和 GORM

### 1. Go 语言特点（对你来说意味着什么）

- **强类型**：每个变量必须声明类型，编译时就能发现拼写错误。对比 Python，你可能会把 `item_sn` 写成 `item_sn2` 而运行时报错，Go 会直接编译不通过。
    
- **结构化**：代码强制按包（package）组织，一个目录一个包，功能边界清晰。
    
- **并发简单**：用 `go func()` 就能开一个轻量级线程，适合需要后台任务（如工资自动计算）的场景。
    
- **编译成单一文件**：最终你只得到一个可执行文件，复制到任何 Linux 机器都能跑，不需要安装任何依赖。
    
- **标准库丰富**：HTTP 服务、JSON 处理、时间操作都内置，不需要 Django 那样的框架。
    

### 2. Gin 框架的作用

- Gin 解决的是“HTTP 请求怎么路由到函数”的问题。它非常轻量，性能极高。
    
- 核心概念：
    
    - **路由（Route）**：把 URL 和函数绑定。如 `r.GET("/api/items", GetItems)`。
        
    - **上下文（Context）**：每个请求都会创建一个 `gin.Context` 对象，你可以从中获取请求参数、设置响应头和返回 JSON。
        
    - **中间件（Middleware）**：在请求到达处理函数之前或之后执行的逻辑，如跨域处理、日志记录、错误恢复。
        

### 3. GORM 是什么

- GORM 是一个 ORM（对象关系映射）库，让你可以用 Go 的 struct（结构体）来操作数据库表，而不用手写 SQL。
    
- 好处：
    
    - 自动建表（AutoMigrate）：你定义好 struct，它会自动生成或更新数据库表结构。
        
    - 链式查询：`db.Where("sn = ?", sn).First(&item)` 比手写 SQL 安全（防注入）且易读。
        
    - 自动处理关联和事务。
        
- 缺点：复杂查询不如 SQL 灵活，但我们可以用原生 SQL 或视图来弥补。

## 第二部分：数据库设计核心思想

### 1. SN 唯一且作为一切操作的锚点

我们以 `items` 表的 `sn` 为主键，这意味着一件物品从入库到报废的整个生命周期，都可以通过 SN 追踪到。

- 采购入库 → SN 首次出现
    
- 拆改加装 → 配件 SN 的状态变为 `used_in_modification`，并记录它被装在了哪台机器 SN 上（`assembled_into_sn`）
    
- 退货 → 原 SN 状态变为 `returned`，如果换新就引入一个新 SN
    
- 返修 → SN 状态变为 `in_repair`
    

**为什么不用自增 ID 做主键？**  
因为真实世界里，我们就是用 SN 来识别实物的，用 SN 做主键自然表达了业务约束（唯一），而且关联查询时不需要再 JOIN 一次获取 SN。

### 2. 产品配置表（products）与物品表（items）的分离

- `products` 存的是“型号模板”，比如“联想 ThinkPad X1 i7-1360P 16G 512G”。它规定了这种配置的标准属性，以及成本参考、预警库存。
    
- `items` 存的是每一个具体的 SN，它虽然也重复了品牌、CPU 等字段，但这是必要的冗余，因为：
    
    - 你可能会扫描到一个 SN，它属于某个产品，但后来产品模板的属性修改了，我们希望这个 SN 的原始属性不变（历史快照）。
        
    - 做库存聚合时，如果用 `products` 表统计库存数量，我们需要 LEFT JOIN 并按照产品属性分组，这样即使某个产品还没有任何 SN，也能显示 0 库存。
        

**冗余的代价**：更新产品属性时，需要同时考虑是否更新已存在 items 的对应字段。通常我们设定 items 的属性在创建时从 products 复制，之后不改动，以保证历史准确性。

### 3. 库存状态与流水

- `items.current_status` 是一个枚举，代表 SN 当前的状态。任何时候查询库存，只需 `WHERE current_status = 'in_stock'`。
    
- `inventory_transactions` 表记录每一次状态变化，这是一个重要的审计日志，可以回答“这个 SN 什么时候被谁改过状态”。在复杂的对账场景中，你可以根据流水还原任意时刻的库存。
    

**为什么不用触发器实现流水？**  
触发器对业务逻辑不透明，后期维护困难。我们选择在 Go 代码中显式写入流水，逻辑更可控。

### 4. 业务单据模式

所有业务表（采购单、出库单、退货单、返修单）遵循“头-行”模式：

- 头部表（如 `purchase_orders`）：记录单据的总体信息（单号、日期、供应商、总额）。
    
- 行表（如 `purchase_inbound_items`）：记录单据中的每一条明细（每个 SN、单价等）。
    

这种模式的好处：

- 字段不乱，一单多行自然扩展。
    
- 方便统计和校验（比如采购校验就是统计行数量与头部预期数量对比）。
    

### 5. 拆改明细的扩展设计

拆改原本只关联 `fulfillments`（出库单），但退货也可能需要拆配件。所以我们在 `disassembly_details` 中增加 `reference_type` 和 `reference_id`，使其能关联多种业务类型。这是设计上的“解耦”，让拆改组件可以复用。

### 6. 工资计算的解耦

工资不直接在拆改时生成，而是事后由“生成工资”功能完成。因为：

- 拆改可能多次修改，工资需要最后确认后才计算。
    
- 工资单价可能调整，如果不解耦，历史工资会变。
    
- 结算周期需要灵活性。
    

因此，`salary_records` 独立存在，生成后即锁定，后续规则变动不影响已生成记录。

## 第三部分：Go 代码架构分层思想

我们采用经典的分层架构，但不是完全照搬 MVC，因为 Go 中更适合按职责分目录：

text

main.go              // 入口，启动服务器
config/              // 读取配置（数据库连接字符串等）
database/            // 初始化数据库连接，自动迁移
models/              // 所有数据库表对应的结构体，以及一些枚举常量
controllers/         // HTTP 处理函数，核心业务逻辑在这里（也可抽 service）
routers/             // 路由定义，将 URL 映射到 controllers 的函数
middleware/          // 跨域等通用处理

### 为什么要分层？

- **单一职责**：每个包只做一件事。`models` 只定义数据结构，`database` 只负责连接和迁移，`controllers` 只处理请求和响应。
    
- **可测试性**：你可以单独测试 `controllers` 中的函数，mock 数据库操作。
    
- **易于修改**：要增加一个 API，你只需在 `routers` 加一行，在 `controllers` 加一个函数，不会影响其他模块。

## 第四部分：核心业务逻辑与算法详解

### 1. 库存产品汇总查询（解决“怎么搜”）

前端搜索栏有多达 7、8 个可选条件，后端如何高效处理？

**算法**：动态构建 GORM 查询链。

go

query := db.Table("products p").
    Select("p.id, p.product_name, ..., COUNT(i.sn) as stock_qty, SUM(i.unit_cost) as total_cost").
    Joins("LEFT JOIN items i ON i.product_id = p.id AND i.current_status = 'in_stock'").
    Group("p.id")
if brand != "" {
    query = query.Where("p.brand = ?", brand)
}
if cpu != "" {
    query = query.Where("p.cpu = ?", cpu)
}
// ... 更多条件

**核心思想**：LEFT JOIN 保证了即使某个产品目前库存为 0，也会显示（因为 products 有记录），只是聚合值为 0。所有过滤条件都添加到查询链上，最后执行 Scan。

**为什么不用 items 直接查？** 因为 items 的 CPU、内存等字段是冗余的且可能不一致，用 products 的标准化字段做聚合更可靠。

### 2. 采购入库校验（数量与 SN 正确性）

逻辑：

1. 前端扫码时，每扫一个 SN 就调用 `POST /purchase-orders/:id/items`，后端会：
    
    - 检查 SN 是否已存在于 `items` 表（唯一性校验）。
        
    - 如果不存在，根据传入的 `product_id` 从 `products` 复制信息，创建一条 `items` 记录，状态设为 `pending`。
        
    - 写入 `purchase_inbound_items` 记录，`is_valid` 默认为 true。
        
2. 用户点击“校验” → `POST /purchase-orders/:id/verify`：
    
    - 统计该采购单下 `is_valid = true` 的行数。
        
    - 与 `purchase_orders.expected_total_qty` 比较，如果相等，`match_result = true`；否则 false。
        
    - 若校验通过（或手动确认后），可以将对应 items 的状态从 `pending` 改为 `in_stock`，并记录入库流水。
        

**为什么分两步？** 这样可以多次扫码、纠错后再统一校验，而且校验结果不会自动改变库存状态，需要人为确认（防止误操作）。

### 3. 出库拆改与成本转移（核心难点）

拆改动作必须保证两点：**配件成本转移**和**库存一致性**。

例子：一台机器 SN = M001，需要加装一块固态 SN = SSD001（成本 200 元），拆除原来的 512G 固态 SN = SSD000（成本 150 元）。

过程：

1. 操作人员扫描 M001，开始拆改。
    
2. 加装 SSD001：
    
    - 在 `disassembly_details` 插入：`{machine_sn: M001, action: 'add', part_sn: SSD001, part_price: 200, staff: '张三'}`。
        
    - 更新 `items`：`SET current_status = 'used_in_modification', assembled_into_sn = 'M001' WHERE sn = 'SSD001'`。
        
    - 记录流水：SSD001 从 `in_stock` 变为 `used_in_modification`。
        
3. 拆除 SSD000：
    
    - 在 `disassembly_details` 插入：`{machine_sn: M001, action: 'remove', part_sn: SSD000, part_price: 150, staff: '张三'}`。
        
    - 更新 `items`：`SET current_status = 'in_stock', assembled_into_sn = '' WHERE sn = 'SSD000'`。
        
    - 记录流水：SSD000 恢复为在库。
        
4. 机器的成本理论上需要重新计算：原机器成本（比如 5000） - 拆下的 SSD 成本（150） + 加装的 SSD 成本（200） = 5050。这个机器成本更新可以在拆改完成后统一计算，也可以实时维护在 `items.unit_cost` 字段中（每次拆改累加/减）。
    

**事务的重要性**：如果加装成功了但配件状态更新失败，库存就对不上了。所以整个拆改动作（插入明细+更新 items）必须放在一个数据库事务中。

### 4. 退货处理与配件回库

退货可能检验出不良品，需要拆下可用配件。流程：

- 退货单关联原 SN。
    
- 检验人员填写检验结果，如果需要拆配件，操作与拆改类似，但 `reference_type='return'`。
    
- 拆下的配件恢复为 `in_stock`（成本保留），报废的配件状态变为 `scrapped`。
    
- 最后原整机/配件状态变为 `returned`（退给供应商）或 `scrapped`。
    

**前后 SN 联动**：如果退货后换了一个新 SN 回来，需要创建新 `items` 记录，状态 `in_stock`，成本可能重新计算（或沿用原价）。原 SN 标记为 `returned`。

### 5. 返修看板与盘点警告

- 返修单创建后，物品状态变为 `in_repair`，`alert_on_stocktake` 设为 true。
    
- 盘点页面需要显示所有 `in_repair` 且 `alert_on_stocktake=true` 的记录，高亮警告。
    
- 修回后，根据处置更新状态，并关闭警告。
    

### 6. 工资计算算法

1. 找到所有 `fulfillments.is_disassembly_completed = true` 且对应 `fulfillment_id` 在 `salary_records` 中不存在（或未生成）的单据。
    
2. 对每个单据，查询其所有拆改明细（`disassembly_details`），按 `staff_name` 分组统计机器台数（每台机器只算一次工资，即使有多条拆改操作）。
    
3. 根据 `fulfillments.type` 找到对应的 `salary_rules.work_type` 和单价，计算每人总工资，写入 `salary_records`。
    
    - 注意：如果一台机器有多个人员操作，可以设计成按人员分别计算（即拆改明细的 `staff_name` 是真正的操作人，每个操作人计算他操作过的机器数）。你的需求里提到“拆改人员，每个人多少钱”，所以可以按人员分组，但可能需要明确：如果多人合作一台机器，是平分还是各自算一台？通常单人单台算，多人合作则拆分。根据业务，可先简单设计为每台机器一个主要拆改人员，由前端填入。

# 库存管理系统全栈设计笔记

> 本笔记整合了基于 Go + Gin + GORM + MySQL + React 的全栈库存管理系统设计，涵盖业务逻辑、数据库、API、前端、技术要点与部署。适合作为学习参考或直接交予AI实现。

---

## 一、项目概况

- **业务范围**：采购入库、SN 扫码校验、入仓/厂直出库（含拆改）、备件库/京东退货（含拆改）、返修配件、工资计算、库存预警与成本核算。
    
- **技术选型**：Go 1.22+ (Gin + GORM) + MySQL 8.0，前端 React (Ant Design)，Docker 部署。
    
- **核心原则**：
    
    - SN 唯一，`items` 表为库存状态唯一真相来源。
        
    - 所有业务动作需同步更新 `items` 状态，并记录流水。
        
    - 成本精确到 SN，拆改工资与业务解耦。

## 二、数据库设计

### 1. 产品配置表 `products`

- 用途：定义标准SKU，用于库存汇总及搜索维度。
    
- 字段：
    
    sql
    
    id INT AUTO_INCREMENT PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL,
    brand VARCHAR(50),
    model VARCHAR(50),
    category ENUM('整机','显示器','内存','固态','显卡','WIFI模块','其他') NOT NULL,
    cpu VARCHAR(50),
    memory VARCHAR(50),
    storage VARCHAR(50),
    gpu VARCHAR(50),
    wifi VARCHAR(50),
    min_stock INT DEFAULT 0 COMMENT '预警下限',
    unit_cost DECIMAL(10,2),
    unit_price DECIMAL(10,2),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
    

### 2. 物品表 `items`

- 用途：每个 SN 一条记录，是库存与状态的唯一真相。
    
- 字段：
    
    sql
    
    sn VARCHAR(50) PRIMARY KEY,
    product_id INT NOT NULL,
    name VARCHAR(100) NOT NULL,
    category ENUM(...) NOT NULL,
    brand VARCHAR(50),
    model VARCHAR(50),
    cpu VARCHAR(50),
    memory VARCHAR(50),
    storage VARCHAR(50),
    gpu VARCHAR(50),
    wifi VARCHAR(50),
    unit_cost DECIMAL(10,2),
    current_status ENUM('in_stock','sold','used_in_modification','returned','in_repair','scrapped','pending') DEFAULT 'in_stock',
    assembled_into_sn VARCHAR(50) NULL COMMENT '若配件被加装，记录机器SN',
    location VARCHAR(50),
    remark TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (product_id) REFERENCES products(id)
    

### 3. 库存流水表 `inventory_transactions`

- 用途：记录每次 SN 变动，便于审计与对账。
    
- 字段：
    
    sql
    
    id INT AUTO_INCREMENT PRIMARY KEY,
    sn VARCHAR(50) NOT NULL,
    transaction_type ENUM('purchase_in','modification_out','modification_in','sale_out','return_in','repair_out','repair_in','scrap_out','other'),
    quantity INT DEFAULT 1,
    reference_type VARCHAR(50) COMMENT '关联业务类型',
    reference_id INT COMMENT '关联业务ID',
    notes TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (sn) REFERENCES items(sn)
    

### 4. 供应商表 `suppliers`

- 字段：`id`, `name`, `contact`, `phone`, `address`, `created_at`
    

### 5. 采购单与扫码明细

- **扫码明细** `purchase_inbound_items`
    
    - 字段：`id`, `purchase_order_id INT`, `sn VARCHAR(50)`, `product_name VARCHAR(100)`, `category VARCHAR(50)`, `unit_price DECIMAL(10,2)`, `scan_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP`, `is_valid BOOLEAN DEFAULT TRUE`, `match_result BOOLEAN NULL`, `created_at`
        

### 6. 出库单（入仓/厂直）`fulfillments`

- 统一管理两类出库，用 `type` 字段区分。
    
- 字段：`id`, `type ENUM('inbound','direct')`, `order_date DATE`, `brand`, `purchase_order_no`, `delivery_center`, `product_code`, `product_name`, `actual_delivery_qty INT`, `ordered_qty INT`, `unit_price`, `ownership`, `channel`, `jd_order_id`, `customer_name`, `customer_phone`, `customer_address`, `tracking_number`, `is_prepared BOOLEAN DEFAULT FALSE`, `is_disassembly_completed BOOLEAN DEFAULT FALSE`, `sn_qty_match BOOLEAN`, `price DECIMAL(10,2)`, `remark TEXT`, `created_at`, `updated_at`
    

### 7. 拆改明细表 `disassembly_details`

- 字段：`id`, `reference_type VARCHAR(20) COMMENT 'fulfillment/return'`, `reference_id INT`, `machine_sn VARCHAR(50)`, `action ENUM('add','remove')`, `part_sn VARCHAR(50)`, `part_name VARCHAR(100)`, `part_price DECIMAL(10,2)`, `quantity INT DEFAULT 1`, `staff_name VARCHAR(50)`, `created_at`
    

### 8. 工资相关表

- **工资规则** `salary_rules`
    
    - 字段：`id`, `work_type ENUM('inbound_modification','direct_modification')`, `unit_price DECIMAL(10,2)`, `description`
        
- **工资记录** `salary_records`
    
    - 字段：`id`, `staff_name`, `fulfillment_id`, `fulfillment_type ENUM('inbound','direct')`, `machine_sn`, `unit_price`, `quantity`, `total_amount DECIMAL(10,2) GENERATED`, `salary_date DATE`, `is_paid BOOLEAN DEFAULT FALSE`, `created_at`
        

### 9. 退货表 `returns`

- 统一备件库、京东、直寄退货。
    
- 字段：`id`, `source_type ENUM('spare_parts','jd_return','direct_return')`, `date DATE`, `brand`, `outbound_order_no`, `spare_barcode`, `sale_return_no`, `jd_order_id`, `purchase_order_no`, `return_order_no`, `original_sn VARCHAR(50)`, `new_sn VARCHAR(50)`, `sku`, `product_name`, `tracking_number`, `applied_qty INT`, `actual_qty INT`, `unit_cost DECIMAL(10,2)`, `inspection_result TEXT`, `handling_method VARCHAR(100)`, `is_completed BOOLEAN DEFAULT FALSE`, `remark TEXT`, `created_at`, `updated_at`
    

### 10. 返修表 `repairs`

- 字段：`id`, `repair_date DATE`, `product_name`, `sn VARCHAR(50)`, `quantity`, `return_address`, `return_tracking_no`, `returned_sn VARCHAR(50)`, `status ENUM('sent','repaired','scrapped')`, `alert_on_stocktake BOOLEAN DEFAULT TRUE`, `disposal_method ENUM('second_hand','scrap')`, `is_completed BOOLEAN DEFAULT FALSE`, `remark TEXT`, `created_at`

### 1. 采购入库与校验

1. 创建采购单（预期数量）。
    
2. 扫码入库：每扫一个SN调用接口，自动写入扫码明细，并同步创建 `items` 记录（若 SN 不存在），状态 `pending`。
    
3. 校验：统计有效扫码数，与预期数量比对，更新 `match_result`；通过后可将物品状态改为 `in_stock`。
    

### 2. 出库拆改（入仓/厂直）

- 出库单可包含多台机器，通过 `fulfillment_items` 记录（可选）。
    
- 拆改操作：
    
    - **加装**：配件状态变更为 `used_in_modification`，设置 `assembled_into_sn`。
        
    - **拆除**：配件恢复 `in_stock`，清空 `assembled_into_sn`。
        
- 必须使用数据库事务，保证拆改明细与物品状态同时更新。
    
- 拆改完成后校验 SN 总数与 `actual_delivery_qty` 是否一致。
    

### 3. 退货处理（含拆改）

- 退货单关联原 SN。
    
- 检验时可进行拆改：复用拆改组件，`reference_type='return'`。
    
- 拆下的配件恢复库存或报废。
    
- 处理完成后更新原 SN 状态，生成新 SN（如换货）并入库。
    

### 4. 返修流程

- 送修：`items` 状态变为 `in_repair`，设置盘库预警。
    
- 修回：填写返回 SN 及处置（二手/报废），更新对应状态，关闭预警。
    

### 5. 工资计算

- 出库单标记“拆改完成”后，通过生成工资接口，根据拆改明细中的操作人员、工资规则计算每人应得工资。
    
- 生成后的工资记录与后续规则变动无关。
    

### 6. 成本核算

- 财务模块查询 `items` 表中特定状态（sold/scrapped...）的 `unit_cost` 总和，可按时间范围过滤。

## 8. 核心业务规则与实现逻辑

### 8.1 采购入库校验

- 扫码时即写入 `purchase_inbound_items`，并创建 `items`（状态 `pending`）。
    
- 校验接口统计该单下有效SN数，与 `expected_total_qty` 比对；若匹配可批量更新状态为 `in_stock`。
    

### 8.2 出库拆改（成本转移）

- 事务：插入拆改明细 → 更新配件状态（`used_in_modification` + `assembled_into_sn` 或恢复 `in_stock`）。
    
- 机器成本实时计算或最终统一更新。
    

### 8.3 退货处理

- 复用拆改表，`reference_type='return'`。
    
- 处理时更新原 SN 状态（`returned`），产生新 SN 则新建物品。
    

### 8.4 工资生成

- 找出已完成且未生成工资的出库单 → 读取拆改明细 → 按人员分组 → 按规则计算金额 → 写入 `salary_records`。
    

### 8.5 成本核算

- 直接查询 `items` 表，按 `current_status` 过滤，`SUM(unit_cost)`，支持按流水表关联时间过滤。

我们理解了用户的反馈：之前的入仓/厂直出库单设计假设每台机器单独记录SN，但实际业务中入仓单会一次性处理多台同配置机器，拆改时也是批量加装配件，不能每台机器都对应每个配件的SN，因为配件可能是一批条码，坏了也是换一批，不可能逐个扫码。同时用户强调采购单是用于出库（和厂直类似的拆改），即采购单实际是出库单的一部分？可能用户想澄清：采购入库（inbound）只是货物进来，而出库（fulfillment）中的入仓和厂直才是需要拆改的部分，而采购单本身只是作为入库记录，并不涉及拆改。需要重新梳理业务。

要求：优化设计，支持批量操作，简化入库扫描流程，使得一次性扫多个SN或整批入库，以及拆改时批量加装配件不强制对应每个配件SN到每台机器。核心是要更贴近实际工作流。

- 工资生成接口读取 `fulfillment_machines` 的数量作为“台数”基础，再结合拆改记录的人员分配计算。

在 GitHub 上，仓库名称是**区分大小写**的，但在实际使用中需要注意几个方面：

---

## 1. GitHub 对大小写的处理

- **创建仓库时**：可以包含大写字母，如 `StockForge`。
    
- **URL 路径**：GitHub 会保留大写，但实际访问不区分大小写（如 `/Yourname/StockForge` 和 `/yourname/stockforge` 都会重定向到正确页面）。
    
- **Git 克隆**：克隆时大小写敏感，`git clone https://github.com/yourname/StockForge.git` 创建的本地文件夹名就是 `StockForge`。
    
- **操作系统差异**：在 Windows/macOS（默认不区分大小写）上，如果两个仓库名仅大小写不同（如 `StockForge` 和 `stockforge`），可能造成本地目录冲突；Linux 严格区分。
    

---

## 2. 大厂的命名惯例

大多数大厂和开源项目遵循 **全小写 + 连字符** 的命名方式，原因如下：

- **兼容性**：URL、命令行、环境变量通常不区分大小写，全小写避免混淆。
    
- **易读性**：连字符分隔单词，如 `stock-forge`。
    
- **一致性**：npm 包名、Docker 镜像名、Go 模块名等都推荐全小写。
    
- **搜索友好**：人们习惯输入全小写来搜索。
    

**常见例子**：

- Google 的 `kubernetes`（非 `Kubernetes`）
    
- Facebook 的 `react`（非 `React`）
    
- Microsoft 的 `vscode`（非 `VSCode`）
    
- 许多内部项目也是 `project-name` 格式。
    

因此，**推荐你的项目仓库名为全小写 + 连字符**：  
建议改为 **`stock-forge`**，而不是 `StockForge`。

---

## 3. Go 模块命名

因为后端使用 Go，模块名也必须遵循 Go 的惯例：

- 全小写，无连字符无下划线（除非必要，但推荐简洁）。
    
- 可以带域名前缀：`github.com/yourname/stockforge`。
    
- 模块名一般与仓库名一致（但 Go 模块路径习惯**不使用连字符**，可以用缩写或驼峰转小写）。  
    例如仓库名 `stock-forge`，模块名可以是 `github.com/yourname/stockforge`（去掉了连字符）。
    

如果你希望仓库名和模块名一致，可以直接取 **`stockforge`**（一个小写单词），这样就完全符合 Go 模块命名风格，也不会有大小写和连字符困扰。

---

## 4. 最终建议

- **GitHub 仓库名**：`stockforge` （全小写，无分隔符）
    
- **Go 模块名**：`github.com/你的用户名/stockforge`
    
- **Docker 镜像名**：`stockforge-backend`、`stockforge-db` 等
    
- **简历中展示**：`StockForge`（首字母大写，作为项目品牌名）
    
- **本地文件夹**：clone 后自动为 `stockforge/`
    

这样既符合大厂惯例，又避免了大小写和分隔符带来的麻烦。
