根据你的业务描述，我整理并优化了数据库设计。下面先给出核心设计思路，然后列出所有表结构，字段已补全，并说明了如何满足前端搜索、库存盘点、拆改工资等需求。

---

## 一、设计思路与优化合并

1. **SN 作为唯一资产标识**  
   所有机器、配件、显示器均以 `sn` 作为全局唯一标识。`items` 表是“物品主表”，一条记录对应一个 SN，记录该物品的固有属性、当前状态、成本等。库存是否在库、去向如何，都通过该表的 `status` 和关联字段体现。

2. **库存查询由视图支撑**  
   前端需要的“按品牌/型号/CPU/内存/固态/WIFI 搜索，显示数量和总价”是**汇总视图**，不是直接查 SN 明细。因此用 `products` 表定义标准配置（SKU），再通过 `items.status = 'in_stock'` 聚合得到每种配置的当前库存数量和总成本。库存预警也放在 `products` 表中。

3. **采购入库简化**  
   设计 `purchase_orders`（采购单头）和 `purchase_inbound_items`（扫码 SN 明细）。总数量校验、SN 正确性校验在后端通过比对 `expected_qty` 与实际扫码数量完成，表里只存结果。

4. **入仓与厂直统一为 `fulfillments` 表**  
   两种业务共用同一张主表，用 `type` 字段区分（`inbound` / `direct`）。特有字段允许为空。拆改明细统一用子表 `disassembly_details`，精确到 SN、价格、动作（加/拆）、操作人。工资根据 `disassembly_details` 关联的 `salary_rules` 或直接记录计算。

5. **备件库退货与京东退货统一为 `returns` 表**  
   通过 `source_type` 区分来源（备件库退/直寄退/京东退货），SN 必须关联。检验结果、处理方式、前后 SN 变更都在此表处理。状态联动通过后端更新 `items` 表实现。

6. **返修配件独立 `repairs` 表**  
   记录送修 SN、返回 SN、维修状态，并可通过 `alert_on_stocktake` 在盘点时高亮提示。修理后出库二手或报废，会同步更新 `items` 状态。

7. **所有变动可追溯**  
   设计 `inventory_transactions` 流水表，记录每次库存变化（入库、出库、拆改消耗、拆改产出、退货、返修等），方便对账。

## 二、数据表详细设计

### 1. 产品配置表 `products`
定义标准 SKU，用于库存汇总和搜索维度。
```sql
CREATE TABLE products (
    id INT AUTO_INCREMENT PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL COMMENT '商品名称',
    brand VARCHAR(50) COMMENT '品牌',
    model VARCHAR(50) COMMENT '型号',
    category ENUM('整机','显示器','内存','固态','显卡','WIFI模块','其他') NOT NULL,
    cpu VARCHAR(50) COMMENT 'CPU型号(整机时填写)',
    memory VARCHAR(50) COMMENT '内存规格(如16G)',
    storage VARCHAR(50) COMMENT '硬盘规格(如512G SSD)',
    gpu VARCHAR(50) COMMENT '显卡型号',
    wifi VARCHAR(50) COMMENT '无线网卡型号',
    min_stock INT DEFAULT 0 COMMENT '库存预警下限',
    unit_cost DECIMAL(10,2) COMMENT '标准成本单价(参考)',
    unit_price DECIMAL(10,2) COMMENT '标准销售单价(参考)',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
) COMMENT '产品配置表，作为库存聚合与搜索的维度';
```

### 2. 物品表（SN 主表）`items`
每个 SN 一条记录，是库存管理的核心。
```sql
CREATE TABLE items (
    sn VARCHAR(50) PRIMARY KEY COMMENT '唯一序列号',
    product_id INT NOT NULL COMMENT '对应的产品配置',
    name VARCHAR(100) NOT NULL COMMENT '物品名称(可沿用产品名，也可自定义)',
    category ENUM('整机','显示器','内存','固态','显卡','WIFI模块','其他') NOT NULL,
    brand VARCHAR(50),
    model VARCHAR(50),
    cpu VARCHAR(50),
    memory VARCHAR(50),
    storage VARCHAR(50),
    gpu VARCHAR(50),
    wifi VARCHAR(50),
    unit_cost DECIMAL(10,2) COMMENT '成本单价',
    current_status ENUM('in_stock','sold','used_in_modification','returned','in_repair','scrapped','pending') DEFAULT 'in_stock' COMMENT '当前状态',
    assembled_into_sn VARCHAR(50) DEFAULT NULL COMMENT '若该配件已被加装到某机器SN，填写机器SN',
    location VARCHAR(50) COMMENT '库位',
    remark TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (product_id) REFERENCES products(id)
) COMMENT '所有资产/物品，以SN为唯一标识，同时记录在库状态和去向';
```

### 3. 库存流水表 `inventory_transactions`
记录所有SN变动，便于追溯。
```sql
CREATE TABLE inventory_transactions (
    id INT AUTO_INCREMENT PRIMARY KEY,
    sn VARCHAR(50) NOT NULL,
    transaction_type ENUM('purchase_in','modification_out','modification_in','sale_out','return_in','repair_out','repair_in','scrap_out','other') NOT NULL,
    quantity INT DEFAULT 1 COMMENT '对于SN唯一，通常为1，但保留字段',
    reference_type VARCHAR(50) COMMENT '关联业务类型，如purchase_order, fulfillment, return, repair',
    reference_id INT COMMENT '关联业务表的ID',
    notes TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (sn) REFERENCES items(sn)
) COMMENT 'SN库存变动流水，用于对账与追溯';
```

### 4. 供应商表 `suppliers`
```sql
CREATE TABLE suppliers (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    contact VARCHAR(50),
    phone VARCHAR(20),
    address VARCHAR(255),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
) COMMENT '供应商信息';
```

### 5. 采购订单与入库明细
采购订单主表 `purchase_orders`：
```sql
CREATE TABLE purchase_orders (
    id INT AUTO_INCREMENT PRIMARY KEY,
    po_number VARCHAR(50) NOT NULL UNIQUE COMMENT '采购单号',
    supplier_id INT,
    order_date DATE NOT NULL,
    expected_total_qty INT NOT NULL COMMENT '采购单上的预期总数量',
    total_amount DECIMAL(12,2) COMMENT '采购总金额',
    status ENUM('pending','partial','completed','cancelled') DEFAULT 'pending',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (supplier_id) REFERENCES suppliers(id)
) COMMENT '采购单主表';
```
入库扫码明细 `purchase_inbound_items`：
```sql
CREATE TABLE purchase_inbound_items (
    id INT AUTO_INCREMENT PRIMARY KEY,
    purchase_order_id INT NOT NULL,
    sn VARCHAR(50) NOT NULL COMMENT '扫码到的SN',
    product_name VARCHAR(100) NOT NULL,
    category VARCHAR(50),
    unit_price DECIMAL(10,2) COMMENT '该SN的单价',
    scan_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    is_valid BOOLEAN DEFAULT TRUE COMMENT 'SN校验是否通过（如格式、重复等）',
    match_result BOOLEAN COMMENT '总数量是否与预期一致，由后端计算后填入',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (purchase_order_id) REFERENCES purchase_orders(id),
    FOREIGN KEY (sn) REFERENCES items(sn)
) COMMENT '采购入库的SN明细，用于校验数量和SN正确性';
```
> 业务校验逻辑：后端根据 `purchase_orders.expected_total_qty` 与同一采购单下有效扫码行数比对，更新 `match_result` 字段。

### 6. 入仓/厂直统一表 `fulfillments`
```sql
CREATE TABLE fulfillments (
    id INT AUTO_INCREMENT PRIMARY KEY,
    type ENUM('inbound','direct') NOT NULL COMMENT '入仓或厂直',
    order_date DATE NOT NULL,
    brand VARCHAR(50),
    -- 入仓字段
    purchase_order_no VARCHAR(50) COMMENT '采购单号（入仓用）',
    delivery_center VARCHAR(100) COMMENT '配送中心（入仓）',
    product_code VARCHAR(50) COMMENT '商品编号',
    product_name VARCHAR(100) NOT NULL,
    actual_delivery_qty INT NOT NULL COMMENT '实际送货数量',
    ordered_qty INT COMMENT '采购单写的数量',
    unit_price DECIMAL(10,2),
    ownership VARCHAR(50) COMMENT '归属',
    -- 厂直字段
    channel VARCHAR(50) COMMENT '途径（厂直）',
    jd_order_id VARCHAR(50) COMMENT '京东订单号（厂直）',
    customer_name VARCHAR(50),
    customer_phone VARCHAR(20),
    customer_address VARCHAR(255),
    tracking_number VARCHAR(50) COMMENT '快递单号',
    is_prepared BOOLEAN DEFAULT FALSE COMMENT '是否备好货（厂直）',
    -- 共用字段
    sn_list_summary TEXT COMMENT 'SN列表简述',
    is_disassembly_completed BOOLEAN DEFAULT FALSE COMMENT '是否完成拆改',
    sn_qty_match BOOLEAN COMMENT 'SN数量是否与送货数量一致',
    price DECIMAL(10,2) COMMENT '最终价格或成本',
    remark TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) COMMENT '入仓单和厂直单统一管理，用type区分';
```

### 7. 拆改明细表 `disassembly_details`
精确记录每台机器的拆改动作，关联使用的配件 SN 及对应成本。
```sql
CREATE TABLE disassembly_details (
    id INT AUTO_INCREMENT PRIMARY KEY,
    fulfillment_id INT NOT NULL,
    machine_sn VARCHAR(50) NOT NULL COMMENT '被拆改的机器SN',
    action ENUM('add','remove') NOT NULL COMMENT '加装还是拆除',
    part_sn VARCHAR(50) NOT NULL COMMENT '配件的SN（加装或拆除的对象）',
    part_name VARCHAR(100),
    part_price DECIMAL(10,2) COMMENT '该配件的成本价',
    quantity INT DEFAULT 1,
    staff_name VARCHAR(50) COMMENT '执行拆改的人员',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (fulfillment_id) REFERENCES fulfillments(id),
    FOREIGN KEY (machine_sn) REFERENCES items(sn),
    FOREIGN KEY (part_sn) REFERENCES items(sn)
) COMMENT '拆改明细，精确到SN、动作、价格、操作人员';
```
> 拆改后的库存变化：加装配件时，配件的 `items.status` 变为 `used_in_modification`，`assembled_into_sn` 填入机器 SN；移除配件则相反。

### 8. 工资计算相关表
拆改工资标准 `salary_rules`（可选，用于管理单台价格）：
```sql
CREATE TABLE salary_rules (
    id INT AUTO_INCREMENT PRIMARY KEY,
    work_type ENUM('inbound_modification','direct_modification') NOT NULL COMMENT '入仓拆改/厂直拆改',
    unit_price DECIMAL(10,2) NOT NULL COMMENT '单台拆改价格',
    description VARCHAR(100),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
) COMMENT '拆改工资单价表';
```
工资记录表 `salary_records`：
```sql
CREATE TABLE salary_records (
    id INT AUTO_INCREMENT PRIMARY KEY,
    staff_name VARCHAR(50) NOT NULL,
    fulfillment_id INT NOT NULL,
    fulfillment_type ENUM('inbound','direct') NOT NULL,
    machine_sn VARCHAR(50) COMMENT '对应拆改的机器SN',
    unit_price DECIMAL(10,2) NOT NULL COMMENT '该台工资单价',
    quantity INT DEFAULT 1 COMMENT '通常一台机器算一次',
    total_amount DECIMAL(10,2) GENERATED ALWAYS AS (unit_price * quantity) STORED,
    salary_date DATE,
    is_paid BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (fulfillment_id) REFERENCES fulfillments(id),
    FOREIGN KEY (machine_sn) REFERENCES items(sn)
) COMMENT '入仓/厂直拆改人员工资记录';
```
> 标识是否需要拆改：入仓和厂直中，标机和显示器通常无需拆改，后端根据 `disassembly_details` 是否关联该单来判定；若未关联，则不生成工资记录。

### 9. 备件库/京东退货处理表 `returns`
统一处理各种退货，关联 SN 及处理结果。
```sql
CREATE TABLE returns (
    id INT AUTO_INCREMENT PRIMARY KEY,
    source_type ENUM('spare_parts','jd_return','direct_return') NOT NULL COMMENT '备件库退/京东仓退/直寄退',
    date DATE NOT NULL,
    brand VARCHAR(50),
    -- 各种单号
    outbound_order_no VARCHAR(50) COMMENT '出库单号（备件库退）',
    spare_barcode VARCHAR(50) COMMENT '备件库条码（备件库退）',
    sale_return_no VARCHAR(50) COMMENT '销退单号（直寄退）',
    jd_order_id VARCHAR(50) COMMENT '订单号（京东/直寄退）',
    purchase_order_no VARCHAR(50) COMMENT '采购单号',
    return_order_no VARCHAR(50) COMMENT '退货单号（京东退）',
    -- 物品信息
    original_sn VARCHAR(50) NOT NULL COMMENT '原始SN',
    new_sn VARCHAR(50) COMMENT '处理后变成的SN（如更换）',
    sku VARCHAR(50),
    product_name VARCHAR(100),
    tracking_number VARCHAR(50) COMMENT '快递单号',
    applied_qty INT COMMENT '申请数量',
    actual_qty INT COMMENT '实际到货数量',
    unit_cost DECIMAL(10,2) COMMENT '成本',
    -- 处理信息
    inspection_result TEXT COMMENT '检验情况（划痕、少件等）',
    handling_method VARCHAR(100) COMMENT '处理方式（如返回供应商、维修、转二手、报废）',
    is_completed BOOLEAN DEFAULT FALSE,
    remark TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (original_sn) REFERENCES items(sn),
    FOREIGN KEY (new_sn) REFERENCES items(sn)
) COMMENT '备件库退货与京东退货统一处理，SN必须关联，处理状态可追踪';
```
> 前后 SN 联动：更新 `returns` 的 `is_completed` 或 `new_sn` 时，后端同步修改 `items` 表中对应 SN 的状态（例如原 SN 变为 `returned`，新 SN 变为 `in_stock`）。

### 10. 返修配件表 `repairs`
```sql
CREATE TABLE repairs (
    id INT AUTO_INCREMENT PRIMARY KEY,
    repair_date DATE NOT NULL,
    product_name VARCHAR(100),
    sn VARCHAR(50) NOT NULL COMMENT '送修SN',
    quantity INT DEFAULT 1,
    return_address VARCHAR(255),
    return_tracking_no VARCHAR(50) COMMENT '寄回单号',
    returned_sn VARCHAR(50) COMMENT '返回的SN（可能相同也可能换新）',
    status ENUM('sent','repaired','scrapped') DEFAULT 'sent' COMMENT '维修状态',
    alert_on_stocktake BOOLEAN DEFAULT TRUE COMMENT '盘库时是否需要明显提示',
    disposal_method ENUM('second_hand','scrap') COMMENT '修理后出库二手/报废',
    is_completed BOOLEAN DEFAULT FALSE,
    remark TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (sn) REFERENCES items(sn),
    FOREIGN KEY (returned_sn) REFERENCES items(sn)
) COMMENT '返修配件表，维修完成可转二手或报废，盘库时警告';
```
> 盘库提示：库存盘点界面可查询 `repairs` 中 `status = 'sent' AND alert_on_stocktake = TRUE` 的记录，高亮展示。

## 三、关于前端搜索与库存预警
- **库存视图**：为方便前端直接使用，可创建数据库视图，例如：
```sql
CREATE VIEW v_inventory_summary AS
SELECT 
    p.id AS product_id,
    p.product_name,
    p.brand,
    p.model,
    p.category,
    p.cpu,
    p.memory,
    p.storage,
    p.gpu,
    p.wifi,
    COUNT(i.sn) AS stock_qty,
    SUM(i.unit_cost) AS total_cost,
    p.min_stock
FROM products p
LEFT JOIN items i ON i.product_id = p.id AND i.current_status = 'in_stock'
GROUP BY p.id;
```
前端可对此视图进行任意组合搜索，并比对 `stock_qty` 与 `min_stock` 实现预警。

## 四、前端链接预留
表结构设计完毕，后端可基于 FastAPI/Django 等提供 RESTful API。前端（网页）只需在局域网内通过 HTTP 访问后端服务。例如：
- 采购入库：`POST /api/purchase-orders` 创建采购单；`POST /api/purchase-orders/{id}/items` 扫码入库
- 库存查询：`GET /api/inventory/summary?search=...`
- 入仓/厂直：`POST /api/fulfillments` 创建记录，`POST /api/fulfillments/{id}/disassembly` 提交拆改明细
- 退货处理：`POST /api/returns`
- 返修：`POST /api/repairs`

API 具体实现不在当前范围内，但表结构已能完全支撑这些操作。

## 优化后的前端接口规划（RESTful API）

### 1. 库存相关（前端最核心的搜索/统计）
- **产品库存汇总（前台搜索用）**  
  `GET /api/inventory/products/summary`  
  参数：`?brand= &model= &category= &cpu= &memory= &storage= &gpu= &wifi= &keyword= &min_stock_alert=true`  
  返回：按产品配置聚合的库存数量、总成本、是否低于预警阈值。  
  对应数据库视图 `v_inventory_summary`。

- **SN 明细列表（点击某个产品下钻）**  
  `GET /api/inventory/items`  
  参数：`?product_id= &status=in_stock &brand= &cpu= &sn= &page= &limit=`  
  返回：分页的单个 SN 记录，用于盘库或追踪。

- **库存全局统计**  
  `GET /api/inventory/statistics`  
  返回：在库总价值、各类别（整机/配件/显示器）数量、库存预警品类数等。

- **库存预警列表**  
  `GET /api/inventory/alerts`  
  返回：所有 `current_stock < min_stock` 的产品配置及缺货数量。

### 2. 采购入库
- **采购单管理**  
  `GET /api/purchase-orders`（列表，可按状态/日期筛选）  
  `POST /api/purchase-orders`（创建采购单，含预期数量和供应商）  
  `GET /api/purchase-orders/{id}`（采购单详情，含预期总数量）

- **扫码入库（SN明细）**  
  `POST /api/purchase-orders/{id}/items`  
  Body：`{ "sn": "...", "product_name": "...", "category": "...", "unit_price": ... }`  
  后端写入 `purchase_inbound_items` 并自动创建对应的 `items` 记录（若 SN 不存在）。

- **入库校验**  
  `POST /api/purchase-orders/{id}/verify`  
  后端计算已扫有效 SN 数量，与采购单预期数量对比，更新 `match_result` 字段，并返回校验结果（一致/短缺/溢出）。

### 3. 入仓/厂直出库与拆改（对应 fulfillments）
- **出库单管理**  
  `GET /api/fulfillments?type=inbound&status= &date=`（入仓）  
  `GET /api/fulfillments?type=direct&status= &date=`（厂直）  
  `POST /api/fulfillments`  
  Body 中包含 type 及其他对应字段（入仓填采购单号等，厂直填订单号、客户信息等）。

- **提交拆改明细**  
  `POST /api/fulfillments/{id}/disassembly`  
  Body：`[{ "machine_sn":"...", "action":"add", "part_sn":"...", "part_price":..., "staff":"..." }, ...]`  
  后端写入 `disassembly_details`，同时更新配件 `items.status` = `used_in_modification`，并将配件的 `assembled_into_sn` 指向机器 SN。

- **标记拆改完成**  
  `PUT /api/fulfillments/{id}/complete`  
  更新 `is_disassembly_completed = true`，并校验 SN 数量是否匹配实际送货数量。

### 4. 工资结算
- **工资规则管理**  
  `GET /api/salary-rules`（查看入仓/厂直单台拆改价格）

- **工资记录**  
  `GET /api/salary/records?staff= &date= &fulfillment_type= &is_paid=`  
  `POST /api/salary/generate`（可选：根据已完成拆改的出库单自动生成工资记录，也可手动逐条创建）

### 5. 退货/备件库处理（统一 returns）
- **退货单管理**  
  `GET /api/returns?source_type=spare_parts&status=`  
  `POST /api/returns`（创建退货单，包含原 SN 等信息）

- **处理退货**  
  `PUT /api/returns/{id}/handle`  
  Body：`{ "inspection_result": "...", "handling_method": "...", "new_sn": "...", "is_completed": true }`  
  后端同步更新 `items` 中对应 SN 的状态（原 SN 变为 `returned`，新 SN 变为 `in_stock` 等）。

### 6. 返修配件
- **返修单**  
  `GET /api/repairs?status=sent&alert=true`（待修回列表，前端可高亮）  
  `POST /api/repairs`（送修登记）  
  `PUT /api/repairs/{id}`（修回更新，填写 `returned_sn`、`disposal_method`，同时更新 items 状态为 `in_stock` 或 `scrapped`）

### 7. 财务成本核算
- **成本汇总**  
  `GET /api/finance/cost-summary?start_date= &end_date= &status=sold,scrapped,used_in_modification`  
  返回：各状态下物品的总成本，可按月份或类别分组。
