# 全栈库存管理系统 · 技术详解

## 第一部分：为什么选择 Go、Gin 和 GORM

### 1. Go 语言特点（对你来说意味着什么）
- **强类型**：每个变量必须声明类型，编译时就能发现拼写错误。对比 Python，你可能会把 `item_sn` 写成 `item_sn2` 而运行时报错，Go 会直接编译不通过。
- **结构化**：代码强制按包（package）组织，一个目录一个包，功能边界清晰。
- **并发简单**：用 `go func()` 就能开一个轻量级线程，适合需要后台任务（如工资自动计算）的场景。
- **编译成单一文件**：最终你只得到一个可执行文件，复制到任何 Linux 机器都能跑，不需要安装任何依赖。
- **标准库丰富**：HTTP 服务、JSON 处理、时间操作都内置，不需要 Django 那样的框架。

### 2. Gin 框架的作用
- Gin 解决的是“HTTP 请求怎么路由到函数”的问题。它非常轻量，性能极高。
- 核心概念：
  - **路由（Route）**：把 URL 和函数绑定。如 `r.GET("/api/items", GetItems)`。
  - **上下文（Context）**：每个请求都会创建一个 `gin.Context` 对象，你可以从中获取请求参数、设置响应头和返回 JSON。
  - **中间件（Middleware）**：在请求到达处理函数之前或之后执行的逻辑，如跨域处理、日志记录、错误恢复。

### 3. GORM 是什么
- GORM 是一个 ORM（对象关系映射）库，让你可以用 Go 的 struct（结构体）来操作数据库表，而不用手写 SQL。
- 好处：
  - 自动建表（AutoMigrate）：你定义好 struct，它会自动生成或更新数据库表结构。
  - 链式查询：`db.Where("sn = ?", sn).First(&item)` 比手写 SQL 安全（防注入）且易读。
  - 自动处理关联和事务。
- 缺点：复杂查询不如 SQL 灵活，但我们可以用原生 SQL 或视图来弥补。

## 第二部分：数据库设计核心思想

### 1. SN 唯一且作为一切操作的锚点
我们以 `items` 表的 `sn` 为主键，这意味着一件物品从入库到报废的整个生命周期，都可以通过 SN 追踪到。
- 采购入库 → SN 首次出现
- 拆改加装 → 配件 SN 的状态变为 `used_in_modification`，并记录它被装在了哪台机器 SN 上（`assembled_into_sn`）
- 退货 → 原 SN 状态变为 `returned`，如果换新就引入一个新 SN
- 返修 → SN 状态变为 `in_repair`

**为什么不用自增 ID 做主键？**  
因为真实世界里，我们就是用 SN 来识别实物的，用 SN 做主键自然表达了业务约束（唯一），而且关联查询时不需要再 JOIN 一次获取 SN。

### 2. 产品配置表（products）与物品表（items）的分离
- `products` 存的是“型号模板”，比如“联想 ThinkPad X1 i7-1360P 16G 512G”。它规定了这种配置的标准属性，以及成本参考、预警库存。
- `items` 存的是每一个具体的 SN，它虽然也重复了品牌、CPU 等字段，但这是必要的冗余，因为：
  - 你可能会扫描到一个 SN，它属于某个产品，但后来产品模板的属性修改了，我们希望这个 SN 的原始属性不变（历史快照）。
  - 做库存聚合时，如果用 `products` 表统计库存数量，我们需要 LEFT JOIN 并按照产品属性分组，这样即使某个产品还没有任何 SN，也能显示 0 库存。

**冗余的代价**：更新产品属性时，需要同时考虑是否更新已存在 items 的对应字段。通常我们设定 items 的属性在创建时从 products 复制，之后不改动，以保证历史准确性。

### 3. 库存状态与流水
- `items.current_status` 是一个枚举，代表 SN 当前的状态。任何时候查询库存，只需 `WHERE current_status = 'in_stock'`。
- `inventory_transactions` 表记录每一次状态变化，这是一个重要的审计日志，可以回答“这个 SN 什么时候被谁改过状态”。在复杂的对账场景中，你可以根据流水还原任意时刻的库存。

**为什么不用触发器实现流水？**  
触发器对业务逻辑不透明，后期维护困难。我们选择在 Go 代码中显式写入流水，逻辑更可控。

### 4. 业务单据模式
所有业务表（采购单、出库单、退货单、返修单）遵循“头-行”模式：
- 头部表（如 `purchase_orders`）：记录单据的总体信息（单号、日期、供应商、总额）。
- 行表（如 `purchase_inbound_items`）：记录单据中的每一条明细（每个 SN、单价等）。

这种模式的好处：
- 字段不乱，一单多行自然扩展。
- 方便统计和校验（比如采购校验就是统计行数量与头部预期数量对比）。

### 5. 拆改明细的扩展设计
拆改原本只关联 `fulfillments`（出库单），但退货也可能需要拆配件。所以我们在 `disassembly_details` 中增加 `reference_type` 和 `reference_id`，使其能关联多种业务类型。这是设计上的“解耦”，让拆改组件可以复用。

### 6. 工资计算的解耦
工资不直接在拆改时生成，而是事后由“生成工资”功能完成。因为：
- 拆改可能多次修改，工资需要最后确认后才计算。
- 工资单价可能调整，如果不解耦，历史工资会变。
- 结算周期需要灵活性。

因此，`salary_records` 独立存在，生成后即锁定，后续规则变动不影响已生成记录。

## 第三部分：Go 代码架构分层思想

我们采用经典的分层架构，但不是完全照搬 MVC，因为 Go 中更适合按职责分目录：

```
main.go              // 入口，启动服务器
config/              // 读取配置（数据库连接字符串等）
database/            // 初始化数据库连接，自动迁移
models/              // 所有数据库表对应的结构体，以及一些枚举常量
controllers/         // HTTP 处理函数，核心业务逻辑在这里（也可抽 service）
routers/             // 路由定义，将 URL 映射到 controllers 的函数
middleware/          // 跨域等通用处理
```

### 为什么要分层？
- **单一职责**：每个包只做一件事。`models` 只定义数据结构，`database` 只负责连接和迁移，`controllers` 只处理请求和响应。
- **可测试性**：你可以单独测试 `controllers` 中的函数，mock 数据库操作。
- **易于修改**：要增加一个 API，你只需在 `routers` 加一行，在 `controllers` 加一个函数，不会影响其他模块。

## 第四部分：核心业务逻辑与算法详解

### 1. 库存产品汇总查询（解决“怎么搜”）
前端搜索栏有多达 7、8 个可选条件，后端如何高效处理？

**算法**：动态构建 GORM 查询链。
```go
query := db.Table("products p").
    Select("p.id, p.product_name, ..., COUNT(i.sn) as stock_qty, SUM(i.unit_cost) as total_cost").
    Joins("LEFT JOIN items i ON i.product_id = p.id AND i.current_status = 'in_stock'").
    Group("p.id")

if brand != "" {
    query = query.Where("p.brand = ?", brand)
}
if cpu != "" {
    query = query.Where("p.cpu = ?", cpu)
}
// ... 更多条件
```
**核心思想**：LEFT JOIN 保证了即使某个产品目前库存为 0，也会显示（因为 products 有记录），只是聚合值为 0。所有过滤条件都添加到查询链上，最后执行 Scan。

**为什么不用 items 直接查？** 因为 items 的 CPU、内存等字段是冗余的且可能不一致，用 products 的标准化字段做聚合更可靠。

### 2. 采购入库校验（数量与 SN 正确性）
逻辑：
1. 前端扫码时，每扫一个 SN 就调用 `POST /purchase-orders/:id/items`，后端会：
   - 检查 SN 是否已存在于 `items` 表（唯一性校验）。
   - 如果不存在，根据传入的 `product_id` 从 `products` 复制信息，创建一条 `items` 记录，状态设为 `pending`。
   - 写入 `purchase_inbound_items` 记录，`is_valid` 默认为 true。
2. 用户点击“校验” → `POST /purchase-orders/:id/verify`：
   - 统计该采购单下 `is_valid = true` 的行数。
   - 与 `purchase_orders.expected_total_qty` 比较，如果相等，`match_result = true`；否则 false。
   - 若校验通过（或手动确认后），可以将对应 items 的状态从 `pending` 改为 `in_stock`，并记录入库流水。

**为什么分两步？** 这样可以多次扫码、纠错后再统一校验，而且校验结果不会自动改变库存状态，需要人为确认（防止误操作）。

### 3. 出库拆改与成本转移（核心难点）
拆改动作必须保证两点：**配件成本转移**和**库存一致性**。

例子：一台机器 SN = M001，需要加装一块固态 SN = SSD001（成本 200 元），拆除原来的 512G 固态 SN = SSD000（成本 150 元）。

过程：
1. 操作人员扫描 M001，开始拆改。
2. 加装 SSD001：
   - 在 `disassembly_details` 插入：`{machine_sn: M001, action: 'add', part_sn: SSD001, part_price: 200, staff: '张三'}`。
   - 更新 `items`：`SET current_status = 'used_in_modification', assembled_into_sn = 'M001' WHERE sn = 'SSD001'`。
   - 记录流水：SSD001 从 `in_stock` 变为 `used_in_modification`。
3. 拆除 SSD000：
   - 在 `disassembly_details` 插入：`{machine_sn: M001, action: 'remove', part_sn: SSD000, part_price: 150, staff: '张三'}`。
   - 更新 `items`：`SET current_status = 'in_stock', assembled_into_sn = '' WHERE sn = 'SSD000'`。
   - 记录流水：SSD000 恢复为在库。
4. 机器的成本理论上需要重新计算：原机器成本（比如 5000） - 拆下的 SSD 成本（150） + 加装的 SSD 成本（200） = 5050。这个机器成本更新可以在拆改完成后统一计算，也可以实时维护在 `items.unit_cost` 字段中（每次拆改累加/减）。

**事务的重要性**：如果加装成功了但配件状态更新失败，库存就对不上了。所以整个拆改动作（插入明细+更新 items）必须放在一个数据库事务中。

### 4. 退货处理与配件回库
退货可能检验出不良品，需要拆下可用配件。流程：
- 退货单关联原 SN。
- 检验人员填写检验结果，如果需要拆配件，操作与拆改类似，但 `reference_type='return'`。
- 拆下的配件恢复为 `in_stock`（成本保留），报废的配件状态变为 `scrapped`。
- 最后原整机/配件状态变为 `returned`（退给供应商）或 `scrapped`。

**前后 SN 联动**：如果退货后换了一个新 SN 回来，需要创建新 `items` 记录，状态 `in_stock`，成本可能重新计算（或沿用原价）。原 SN 标记为 `returned`。

### 5. 返修看板与盘点警告
- 返修单创建后，物品状态变为 `in_repair`，`alert_on_stocktake` 设为 true。
- 盘点页面需要显示所有 `in_repair` 且 `alert_on_stocktake=true` 的记录，高亮警告。
- 修回后，根据处置更新状态，并关闭警告。

### 6. 工资计算算法
1. 找到所有 `fulfillments.is_disassembly_completed = true` 且对应 `fulfillment_id` 在 `salary_records` 中不存在（或未生成）的单据。
2. 对每个单据，查询其所有拆改明细（`disassembly_details`），按 `staff_name` 分组统计机器台数（每台机器只算一次工资，即使有多条拆改操作）。
3. 根据 `fulfillments.type` 找到对应的 `salary_rules.work_type` 和单价，计算每人总工资，写入 `salary_records`。
   - 注意：如果一台机器有多个人员操作，可以设计成按人员分别计算（即拆改明细的 `staff_name` 是真正的操作人，每个操作人计算他操作过的机器数）。你的需求里提到“拆改人员，每个人多少钱”，所以可以按人员分组，但可能需要明确：如果多人合作一台机器，是平分还是各自算一台？通常单人单台算，多人合作则拆分。根据业务，可先简单设计为每台机器一个主要拆改人员，由前端填入。

## 第六部分：Go 代码中的关键技术细节

### 1. 模型定义与 GORM 标签
```go
type Item struct {
    SN             string `gorm:"primaryKey;size:50"`
    ProductID      uint
    CurrentStatus  string `gorm:"column:current_status;size:30;default:'in_stock'"`
    // ...
}
```
`gorm` 标签指定了数据库列名、类型、约束等。通过 `db.AutoMigrate(&Item{})` 即可生成表，无需手动 SQL。

### 2. 枚举的处理
Go 没有内置枚举，我们定义常量：
```go
const (
    StatusInStock = "in_stock"
    StatusSold    = "sold"
    // ...
)
```
在模型中用 string 字段存储，代码中使用常量，避免字符串拼写错误。

### 3. 请求验证与绑定
Gin 的 `ShouldBindJSON` 可以将请求体自动绑定到 struct，并通过 `binding` 标签验证必填：
```go
type CreatePurchaseInput struct {
    PONumber string `json:"po_number" binding:"required"`
    // ...
}
```
如果缺少必填字段，会返回 400 错误。

### 4. 事务处理
```go
tx := db.Begin()
// 创建拆改明细
if err := tx.Create(&detail).Error; err != nil {
    tx.Rollback()
    return
}
// 更新物品状态
if err := tx.Model(&Item{}).Where("sn = ?", partSN).Updates(...).Error; err != nil {
    tx.Rollback()
    return
}
tx.Commit()
```
一旦任何一步出错，整个事务回滚，保证数据一致性。

### 5. 跨域中间件
```go
func Cors() gin.HandlerFunc {
    return func(c *gin.Context) {
        c.Header("Access-Control-Allow-Origin", "*")
        c.Header("Access-Control-Allow-Methods", "GET, POST, PUT, DELETE, OPTIONS")
        if c.Request.Method == "OPTIONS" {
            c.AbortWithStatus(204)
            return
        }
        c.Next()
    }
}
```

### 6. 配置管理
可以用环境变量或配置文件（如 `config.yaml`），在 `config` 包中读取并提供给整个项目。

后端连接数据库的 DSN 中主机名是 `db`（服务名），Docker 内部网络会自动解析。
