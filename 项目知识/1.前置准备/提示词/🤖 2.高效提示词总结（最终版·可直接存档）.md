## 🤖 高效提示词总结（最终版·可直接存档）

> 这份总结来源于你今晚完整的实战对话。  
> 你从一个模糊需求出发，通过不断约束我的输出，最终得到了一份**结构清晰、可直接复制、永久可用**的 MySQL 部署手册。  
> 以下技巧适用于任何 AI 工具，加上它们就能获得类似质量的结果。

---

### 一、核心原则

| 原则 | 说明 |
|------|------|
| **给结构** | 明确要求“先正常流程，后故障排查”，分层输出 |
| **给边界** | 指定平台（Win）、起点（只有 Docker）、终点（VS Code 的 py 能连上） |
| **给参数** | 强制使用实际密码（如 `123456`），禁止占位符 |
| **给格式** | 要求注释表、速查表，方便阅读和存档 |
| **给用途** | 说明“直接复制到 Obsidian 永久保存”，去除水分 |

---

### 二、提示词技巧分类

#### 1. 明确输出结构
- “你标明那些是没问题一步就能执行完成，哪些是错误后的排查”
- “你先全流程，后排查”
- “分开写，注重你的格式，需要人看的懂”

#### 2. 用“一条命令”压缩冗余
- “命令启动 docker 的 mysql，一条命令全设置”
- “你给我整理一份最终的命令，我要实现全部功能的”

#### 3. 要求参数注释表
- “后面带个注释表”
- “命令参数注释表”
- “有明确的注释”

#### 4. 限定环境和范围
- “win电脑”
- “从电脑只有 docker 一直到 vs 的 py 能链接为止”
- “白电脑”

#### 5. 要求排错手册
- “把所有命令，注释，检查，排查，都需要写全”
- “常见错误排查速查表”

#### 6. 使用已验证的成功配置
- “现在我的密码是 '123456' 你把上面全流程总结”
- “我执行完这个就成功了，现在你需要再重新完整的总结一下”

#### 7. 指定输出用途
- “我要直接复制到 obsidian 里面，我直接存下了，以后永远直接能用”

#### 8. 追问原理（可选）
- “现在可以给我讲一下我的为什么可以链接了吗”

---

### 三、万能提示词模板

以后你只需要填空即可：

```
你现在给我整理一份【全流程】的[主题]手册。
要求：
1. 从[初始状态]到[最终目标]，一条命令搞定所有配置；
2. 分开写：先【正常流程】（复制即用），后【故障排查】（只列出真正会遇到的错误和解决命令）；
3. 每个命令都要带【参数注释表】，方便看懂；
4. 所有密码/配置参数用我给的[实际值]，不要用占位符；
5. 格式清晰，可以直接复制到[Obsidian]永久保存；
6. 最后附上成功原理的简要解释。
```

### 四、高级用法（不同场景的变体）

#### 📦 需要可执行脚本时
```
给我一条命令，实现[功能]，需要带所有参数，并且把参数写成注释表，可直接复制运行。
```

#### 🐛 踩坑后要求固化
```
我刚才碰到了[错误A]、[错误B]，现在成功了，你把全流程重新总结，包括这些错误的排查，做成速查表。
```

#### 📚 需要存档笔记时
```
输出一份完整手册，我要直接复制到 Obsidian，永久使用。格式要清晰，有正常流程、故障排查、注释表。
```

#### 🤯 快崩溃时
```
我现在已经非常累了，不想搞了，你重新给我一个能用的[主题]全流程，从[起点]到[终点]，一条命令搞定，别废话。
```

### 五、如何把以上技巧整合成一个“系统指令”

如果你希望以后每次提问都自动应用这些约束，可以在对话开头加上：

```
从现在开始，你输出的所有技术方案必须满足：
1. 分正常流程和故障排查两部分；
2. 命令必须可一键执行，附带参数注释表；
3. 使用我提供的实际配置值，不要占位符；
4. 格式清晰，可直接复制到 Obsidian 永久存档；
5. 只写真正会遇到的问题，不要无意义的警告。
```
