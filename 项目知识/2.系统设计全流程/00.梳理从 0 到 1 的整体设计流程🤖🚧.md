### 3. 系统架构设计（前后端分离 + Docker）
- **整体架构**：  
  浏览器 → Nginx（前端静态资源 + 反向代理） → Go API 服务 → MySQL  
  也可以开发阶段前端 dev server 直连后端，生产用 Nginx 容器统一入口。
- **容器划分**：
  - `frontend`：Nginx 提供静态文件，可配置反代 `/api` 到后端。
  - `backend`：Go 应用，暴露 `:8080`。
  - `db`：MySQL 容器，数据用 named volume 持久化。
- **注意**：
  - 使用 Docker 网络让容器通过服务名通信（例如 `backend:8080`）。
  - 前后端分离后必须处理好 CORS，开发时允许前端域名，生产时收紧。

### 4. 数据库设计（重点：“分的不能再分”）
你提到的“分的不能再分”实际指的是**数据库范式理论**，通常要做到 **第三范式（3NF）** 甚至 **BCNF**：
- **第一范式（1NF）**：列不可再分，每列都是原子值（例如地址不要存成 `"北京,海淀"` 这种复合值）。
- **第二范式（2NF）**：满足 1NF，且每个非主键列完全依赖于主键（消除部分依赖）。
- **第三范式（3NF）**：满足 2NF，且非主键列不传递依赖主键（例如表中有 `部门ID`，就不要同时存 `部门名称`，除非是性能需求）。
- **“分的不能再分”**：即每一个表只描述一个实体或关系，字段之间没有冗余的依赖。像经典例子：`学生表`、`课程表`、`选课表`，将多对多关系拆为独立表。

**设计步骤：**
1. **概念设计**：画 ER 图，提取实体（用户、订单、商品等）和关系。
2. **逻辑设计**：转换为表，严格按照 3NF 拆分，避免数据冗余与更新异常。
3. **物理设计**：
   - 选择合理数据类型（金额用 `DECIMAL`，状态用 `TINYINT` 或 `ENUM`）。
   - 建立索引（高频查询字段、唯一约束、联合索引的“最左前缀”原则）。
   - 必要时设置外键（明确约束，但生产可能因性能或分库而不用外键，靠应用层保证）。
4. **迁移管理**：使用 `golang-migrate`、`goose` 等工具，所有表结构变动用版本化 SQL 文件记录。

**注意事项（平衡范式与性能）**：
- 严格遵循高范式会拆出很多表，导致多表 Join 影响查询性能。
- **适当的反范式化**：如订单表中同时保存“商品快照名称”、“单价”，以减少历史价格变动的影响，这是有意冗余，不是设计缺陷。
- 先正规化，再根据**实际查询热点**和性能测试结果做有限反范式处理。

### 5. API 设计（前后端契约）
- **风格**：RESTful，资源名用名词复数（`/api/v1/users`）。
- **统一响应格式**：
  ```json
  {
    "code": 200,
    "message": "success",
    "data": { ... }
  }
  ```
- **认证**：除了登录/注册，其余接口需要 JWT，通过中间件校验。
- **分页与过滤**：`?page=1&page_size=20&sort=-created_at&status=active`
- **版本控制**：URL 路径版本 `/v1/`，避免破坏性变更影响客户端。
- **文档先行**：先用 Swagger 注解定义 API，前后端同步开发，减少联调摩擦。

**注意**：
- 敏感操作（删除、修改他人资源）必须验证权限，不能仅靠前端隐藏按钮。
- 设计时可产出 OpenAPI 文件，直接导入 Postman 或自动生成前端请求代码。

### 7. 开发环境搭建与 Docker 集成
- **开发阶段**：
  - 后端用 `air` 实现热重载，挂载源码目录到容器，修改即生效。
  - 前端 dev server 通过代理将 `/api` 请求转发到后端容器。
- **`docker-compose.yml` 结构**：
  ```yaml
  services:
    db:
      image: mysql:8.0
      environment:
        MYSQL_ROOT_PASSWORD: secret
        MYSQL_DATABASE: myapp
      volumes:
        - db_data:/var/lib/mysql
    backend:
      build: ./backend
      ports: "8080:8080"
      depends_on: db
      environment: ...
    frontend:
      build: ./frontend
      ports: "80:80"
  volumes:
    db_data:
  ```
- **注意**：`.env` 文件管理环境变量，不要将密码硬编码；`Dockerfile` 使用多阶段构建减小镜像体积（Go 应用最终用 `scratch` 或 `alpine`）。

### 8. 前后端联调与测试
- **Mock 接口**：后端未完成时，前端可先用 Mock 数据或工具（如 Apifox）模拟。
- **测试金字塔**：
  - 单元测试：Go 用 `testing` 包 + `testify`；前端用 Vitest/Jest。
  - 集成测试：用 `dockertest` 启动真实 MySQL 测试存储层。
  - 端到端测试：Playwright/Cypress 模拟用户操作。
- **CORS 处理**：后端明确设置允许的前端域名、允许的 header 和 methods，开发阶段可宽松，生产从严。

### 9. 安全性要点
- **SQL 注入**：使用参数化查询（`?` 占位符）或成熟的 ORM，绝不可拼接 SQL。
- **密码存储**：`bcrypt` 哈希，不存明文。
- **JWT 安全**：短有效期的 Access Token（15min），配合 Refresh Token 轮换；退出时失效 Refresh Token。
- **HTTPS**：生产环境必须使用 TLS，可以通过 Nginx 反向代理或云服务商证书完成。
- **敏感配置**：通过环境变量或 Docker secrets 注入。

---

### 10. CI/CD 与部署
- **CI 流程**：代码推送 → 运行测试 → 构建 Docker 镜像 → 推送到镜像仓库（Docker Hub、阿里云等）。
- **CD 流程**：在目标服务器拉取最新镜像，执行 `docker-compose up -d`，或使用 Watchtower 自动更新。
- **数据库迁移**：部署时自动执行迁移脚本（比如在容器启动脚本中调用 `migrate up`）。
- **注意**：先备份数据库再执行迁移；滚动更新时考虑向后兼容的字段变更（先加列再删列）。

---

### 11. 监控、日志与运维
- **日志**：应用日志输出到 `stdout`/`stderr`，Docker 收集，使用 `docker logs` 或集中式 EFK/Loki 栈查看。
- **健康检查**：后端提供 `/health` 端点，Docker Compose 可配置 `healthcheck`。
- **指标监控**：嵌入 Prometheus 端点，可视化请求延迟、错误率、DB 连接数等。
- **数据库备份**：定时 `mysqldump` + 同步到对象存储。

---

### 贯穿全程的核心注意事项
1. **数据库“分”的艺术**：开始时严格走 3NF，但当某查询频繁 JOIN 超过 3~4 张表且数据量大时，考虑部分反范式或汇总表优化，别把“分的不能再分”变成性能杀手。
2. **接口设计先于开发**：前后端共同确认 API 契约（路径、参数、返回值），减少集成阶段的混乱。
3. **环境一致性**：用 Docker 封装所有服务，从开发到生产一套 compose 模板（通过不同 `.env` 区分）。
4. **安全性左移**：从设计之初就考虑权限模型、输入校验，不要最后再补。
5. **版本化一切**：代码用 Git，数据库用迁移工具，API 用版本路径。

## 一、必要文档（分析与设计的产物）

这些文档不一定要长篇大论，但要结构清晰、够用。

### 1. 产品需求文档（PRD）
- **内容**：用户角色、功能列表、用户故事、页面流程图、验收标准。
- **一人项目简化**：用**按功能拆分的用户故事 + 验收标准**即可，可放在项目 Wiki 或 Git 的 `docs/prd/` 下。
- **注意**：每个功能要说清楚“谁、做什么、为什么”，并标注优先级。

### 2. 系统设计文档（概要设计 + 详细设计）
- **概要设计**：系统架构图（容器、服务、网络）、技术选型理由、模块划分。
- **详细设计**：关键流程时序图（如登录流程、下单流程）、重要算法说明、状态机图。
- **产出**：架构图可用 Draw.io 导出图片，文字说明写入 `docs/design/architecture.md`。

### 3. API 接口文档（契约文档）
- **工具**：Swagger / OpenAPI 规范，直接由 Go 代码注解生成在线文档（如 `swaggo/swag`）。
- **标准**：统一响应格式、错误码枚举、每个接口的请求/响应示例。
- **必须覆盖**：认证方式（JWT）、分页规范、文件上传规范。
- **存放**：可生成静态文件部署，也可始终维护 `docs/api/openapi.yaml`。

### 4. 数据库设计文档
- **必要内容**：
  - ER 图（用 PlantUML / Draw.io 或 DBeaver 导出）。
  - 数据字典：每张表的每个字段的含义、类型、长度、是否必填、默认值、约束。
  - 索引说明：索引名、字段、类型、用途。
  - 数据关系说明：外键逻辑（哪怕未用物理外键）。
- **存放**：`docs/database/er.md` 和 `docs/database/data-dictionary.md`。

### 5. 部署与运维文档
- **内容**：
  - 环境依赖清单（Go版本、MySQL版本等）。
  - Docker 部署步骤、环境变量说明、健康检查地址。
  - 备份策略与恢复方法。
  - 日志与监控接入方式。
- **存放**：`docs/deployment/operations.md`。

### 6. 测试文档（测试用例与报告）
- **必要**：
  - 核心接口测试用例表（输入、期望输出、覆盖边界）。
  - 关键业务流程端到端测试用例。
  - 可以在代码目录中用 `_test.go` 和前端 `*.spec.ts` 体现，但最好有简要的覆盖说明文档 `docs/testing/test-plan.md`。

> 以上文档，一个人开发时可以都放在 Git 仓库的 `/docs` 目录下，用 Markdown 维护。版本历史即是文档演化记录。

## 二、必要文件（项目运行与建设的最小集合）

以下是**代码之外**的配置文件、数据文件和构建文件，缺一不可。

### 1. 数据库迁移文件（SQL 文件）
- **路径**：`migrations/` 目录，如：
  ```
  migrations/
    000001_create_users_table.up.sql
    000001_create_users_table.down.sql
    000002_add_orders_table.up.sql
    000002_add_orders_table.down.sql
  ```
- **标准**：使用 `golang-migrate` 或 `goose` 的版本化命名；每次变更一个文件对；up 是正向迁移，down 是回滚。
- **注意**：禁止手动改库后用工具导出，必须手写迁移并纳入版本控制。

### 2. Docker 相关文件
- **`Dockerfile`**（后端和前端分别一个）：
  - Go 后端：多阶段构建，最终基于 `scratch` 或 `alpine`，暴露端口。
  - 前端：构建产物用 Nginx 镜像，并包含 Nginx 配置文件。
- **`docker-compose.yml`**（根目录）：
  - 定义 `db`、`backend`、`frontend` 服务，声明网络和卷。
- **`.dockerignore`**：排除 `node_modules`、`.git`、本地编译产物等。

### 3. 环境配置模板与变量管理
- **`config/config.example.yaml`** 或 `.env.example`：
  - 列出所有可配置项及示例值（数据库连接、JWT密钥、日志级别等），**绝不包含真实密码**。
- 实际环境变量通过 `.env`（不提交 Git）或 Docker 环境变量注入。

### 4. 后端项目核心文件
- `cmd/server/main.go`：入口，加载配置、初始化日志、启动服务。
- `internal/` 下分层 handler、service、repository、model。
- 配置文件解析逻辑（如 Viper）。
- 迁移工具执行入口（可在 `cmd/migrate/main.go` 中集成）。
- `go.mod` 和 `go.sum`。

### 5. 前端项目核心文件
- `package.json`、构建脚本。
- `vite.config.ts`（代理配置、构建配置）。
- `src/api/` 下封装了所有后端接口的请求模块（遵循接口文档）。
- `src/router/` 路由守卫与权限。
- Nginx 配置文件 `nginx.conf`（前端容器内使用，处理 SPA 路由和反向代理 `/api`）。

### 6. CI/CD 配置文件
- 如 `.gitlab-ci.yml`、`.github/workflows/deploy.yml` 等，定义测试、构建镜像、部署流程。
- 即便一个人，用 CI 自动化构建也能省去大量重复劳动。

### 7. 项目 README.md
- 项目简介、快速启动（`docker-compose up`）、目录结构、开发说明、文档索引。

### 8. 其他静态配置文件
- `.gitignore`（合并 Go、Node、Docker 的忽略规则）。
- `.editorconfig`（保证编码风格一致）。
- 许可证文件（若开源）。
